## 0. お題（ユーザー原文の要旨）

## 1. コードで裏取り済みの現状（2026-07-04 HEAD `b3337de`）

### クライアント測位（遅さ・失敗の主犯候補）
- `lib/get-current-location.ts` の `getCheckinLocation`: Web は `watchWebLocationBestSample` で
  **PC最大45秒 / モバイル最大22秒**待つ。目標精度 PC30m/モバイル25m。妥協条件は
  「4秒改善なし かつ acc≤120m」or「80%経過 かつ acc≤250m」。
- PCのWi-Fi/IP測位は acc 1000〜10000m超になりやすく、妥協条件にすら届かず
  **ハードタイムアウトまで45秒待って失敗**が起きうる。
- Native は高精度GPS→acc>80mなら**もう1回GPS**（＝2回分待つ）。
- サーバーは **acc>10000m を BAD_REQUEST で拒否**（`checkin-guards.ts` `isAcceptableAccuracy`）
  → PCのIP測位はこれに引っかかる＝「失敗する」の有力原因。
- 既にある資産:
  - 位置許可ワームアップ `lib/geolocation-warmup.ts`（enableHighAccuracy:false の事前取得）
  - 高精度アンカー補正（スマホで記録した直近足あとでPCの粗測位を補正、`refineDesktopCheckinLocation`）
  - **軽量測位パターン `lib/live-location-session.ts`（速いキャッシュ→watch→500ms後に高精度1発）
    がライブ位置共有用に存在するがチェックイン未適用**
  - チェックイン画面には既に locating/saving の2フェーズ、保存前プレビュー(adjust)で
    地図クリック補正・再測位、PC向け注意文言がある（`checkin-authenticated-screen.tsx`, 1238行）

### サーバー `encounter.checkIn`（`modules/encounter/api/encounter.ts`）
1. 入力検証（acc>10000拒否 / 座標検証）
2. `getUserSettings` + `reverseGeocodeWithTimeout(2.5s)` を並列 —
   **Nominatim逆ジオコーディングが INSERT のクリティカルパスに直列**（表示用データなのに保存をブロック）。
   Nominatim側: throttle付き・8s abort・429/5xxでクールダウン・座標丸めキャッシュあり。
3. `insertLocation` + `upsertVisitedArea` 並列
4. `getMostFrequentNightH3R8` → homeMaskCell 更新（既に fire-and-forget 非同期）
5. マッチング用クエリ4本並列（nearby / timeshift / blockSet / todayPairSet）
6. マッチ分の `insertEncounterIfNew` を**逐次ループ**
- DB は Vercel Function → Railway **外部プロキシURL**へ都度接続＝リージョン跨ぎ往復が約9回以上＋コールドスタート。
- `saved:false` を返す失敗パスあり（INSERT失敗時）。

### 症状との対応（推定確度: 高）
- 「失敗」= PC acc>10000 拒否 / 位置許可拒否 / 45秒タイムアウト
- 「遅い」= 測位待ち（最大45秒）+ geocode 2.5s + DB多往復 + コールドスタート

## 2. 会議ハーネスの見解（要旨）

### 合意された方向（3頭脳が同型の提案）
- **段階的受理**: 粗測位で即 `provisional` 保存 → 高精度が取れたら同一レコードを confirm/PATCH。
  acc>10000 でも「拒否せず受けて後から直す」。
- **サーバー非同期化**: 逆ジオコーディングとマッチングを応答のクリティカルパスから外す
  （即200を返し、市区町村名は後追い補完、マッチングは非同期→結果はプッシュ/次回表示）。
- **楽観UI**: 押した瞬間に成功体験を見せ、裏で確定。二段階トースト
  （「チェックインしました！」→「位置が確定しました」/「精度が不十分です。再取得します」）。
- **オフライン/リトライキュー**: UUID付きFIFO（上限・重複防止）。

### 批判役の重要警告（設計で必ず応答すること）
1. **再マッチング整合性が最大の罠**: 粗い位置でマッチング済み→高精度上書きでセルが変わったら?
   相手側の封筒履歴は修正されず矛盾する。「provisionalではマッチングしない」or
   「同一セル確定時のみ即マッチ、変動時は再計算キュー」等の明確な規約が必要。
2. **楽観UIのロールバックUX**: 実は失敗していた時に信頼を失う。失敗を隠さない二段階表示・
   再試行導線を最初から設計に入れる。
3. **低精度データの汚染**: マッチング品質・軌跡マップ表示・「正確な足あと」の価値を毀損しない
   ように、provisional の可視性/マッチング参加条件を分離する。
4. **レース/二重**: 同時多重チェックイン・キュー再送の重複（クライアント生成の冪等キー推奨）。
5. **過剰演出**: アニメで遅く感じさせない。演出は「待ち時間の隠蔽」に使う（測位中の演出）。
6. 位置偽装は既知の未対策領域（今回のスコープ外にするなら明記）。

### 個別に面白かった発散案
- 「精度改善の見込み確率」を出して見込み薄なら早期に妥協确定（ダラダラ待たない）。
- 過去1週間の高精度履歴クラスタリングでIP測位を補正（既存アンカー補正の一般化）。
- visitedAreas 更新をバッチ化して往復削減。

## 3. 地雷マップ（過去に壊した箇所・確定済みの制約・作法）

1. **挙動契約（壊すな）**: 正確な lat/lng/accuracyM 保存・永続（丸め座標での妥協不可）/
   h3R8+500mグリッドのマッチング互換 / 同日同ペア重複 encounter 抑止 / ブロック・停止ユーザー除外。
   `__tests__/encounter-core.test.ts`（71件）と checkin-guards のテストが既存の安全網。
2. **DBスキーマ変更は Stop And Ask**（refactor-instructions）。provisional フラグ等の
   カラム追加は設計に含めてよいが、実装前にユーザー承認が必要と明記すること。
3. **`__tests__/get-current-location.test.ts` はユーザーの未コミット差分でパース破損中**。
   このファイルを上書き・削除しない。新テストは別ファイルに。
4. **UI/UX は `DESIGN.md` 遵守**。最重要コピー「会いたい君がいる現在地」。`theme/tokens/` を使う。
   汎用SaaS風・抽象グラデーション・カード入れ子・正確な場所が見えない地図UIを避ける。
5. **新地雷（2026-07-04確定）: `import()` したチャンクから React コンポーネントを直接描画して
   常時マウントするの禁止**（本番全面凍結OOMの真因だった。`lazy()`+Suspense は可）。
   詳細 docs/investigation/auth-home-oom-root-cause.md。
6. **無限アニメは cleanup + タブ非フォーカス停止が必須**（expo-router Tabs はタブ切替で
   アンマウントしない）。演出追加時は `useReducedMotion` と `animate` prop の作法に従う
   （envelope-pulse.tsx が手本）。
7. **認証フロー（use-auth / Clerk Satellite / oauth）と、ゲスト `/` の tRPC defer 境界は触るな。**
8. **Nominatim は 1req/s throttle・429クールダウン実装済み**。呼び出し頻度を増やす設計は不可
   （非同期化・キャッシュ強化はOK）。
9. **パフォーマンス施策は「本番ビルド+凍結プローブ」ゲートを通す**（7/1のPageSpeed施策が
   全面障害を生んだ教訓）。実装フェーズの検証要件に含める。
10. 過去の設計判断（[[surechigai-onetap-redesign-spec]]）: 位置許可はユーザー操作起点。
    勝手に権限プロンプトを出さない。
11. Vercel/GitHub Actions は無料枠維持（常駐ワーカー等のコストが乗る案は要注記）。
    DB は Railway 有料契約済み。sweep 用 GitHub Actions（15分間隔）が既存の非同期実行手段。

## 4. 設計が答えるべき問い（司令塔からFableへ）

1. 測位戦略: 「即時に受けて、必要なら後から良くする」を、正確な足あとという製品価値と
   両立させる具体的な状態機械（provisional→confirmed の条件、マッチング参加条件、表示規約）。
2. サーバー応答: 体感1秒のために checkIn 応答に何を残し、何を後回しにするか。
   後回しの実行基盤は何か（応答後の waitUntil / sweep GHA / 次回リクエスト時 lazy 等、無料枠内で）。
3. 失敗ゼロ化: 権限・精度・ネットワークの3大失敗をそれぞれどう飲み込むか。
4. 楽しさ: チェックイン瞬間の演出・報酬（封筒期待感/エリア名/連続記録/切手帳との接続）を、
   遅く感じさせず・アニメ地雷を踏まずにどう作るか。DESIGN.md のトーンで。
5. 段階導入計画: 一撃の大改修ではなく、効果の大きい順に安全に出す順序
   （例: Phase1=クライアント測位の高速化だけで体感が激変するはず）。
