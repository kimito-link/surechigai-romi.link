## 2. 会議ハーネスの見解（3頭脳・要旨）

### 批判役 groq/gpt-oss-120b — 「対症療法を並べるな、観測データを先に取れ」
- A〜Fは相互排除できておらず、現状情報では主因を判別**できない**。推測対処はアンチパターン。
- **最優先は Sentry DSN 有効化**（実装済みなのだから設定するだけ）。OOM前後のメモリ・クエリ数・レンダ回数を取る。
- SW は「手動アンインストール用スクリプト配布」で B 仮説を即検証可能。
- React Query 永続化を一時無効化して localStorage サイズを確認する即席テスト。
- Clerk 認証確定後のレンダ回数を console 計測して C 仮説を定量化。
- 認証済みユーザーを lite 版へ強制リダイレクトする別エントリポイント案。
- リスク警告: キャッシュパージ/SW強制更新は全ユーザー一斉リロードを誘発。UI最小化だけではレンダリングロジックが走り続けるので減らない。

### 発散役 groq/qwen3.6-27b — 「本命は B: SW×chunk-recover の雪だるま」
- **B が最優先**。デプロイ直後悪化・「ちかちか」・OOM（リロード横断のメモリ蓄積）を全部説明できる。
- F（無限ループの誤表示）は独立仮説ではなく **B/C/E の発現メカニズム**。
- A(SVG700要素)は presence 無効でも残る点・デスクトップでは通常OOMしない点から主犯ではなく増悪因子。
- 具体案: chunk-recover に**ハードなリロード回数上限**（例: 3回で停止+console記録）、SW unregister テスト、DevTools Memory の allocation timeline で Detached DOM を見る。
- 非常口案: 認証済みルートで SW を無効化する / `?sw=disable` 相当のバイパス。

### 発散alt groq/gpt-oss-20b — 「本命は C: Clerk確定待ち再レンダ嵐」
- 「一度レンダ→ロード継続→死」は、認証確定でクエリ enabled が一斉 true 化→再レンダ連鎖の形に合う。
- ただし自認リスク: C固執は危険、AとBを並行検証すべき。
- 止血の優先順: SW強制更新(b) → 認証済みホーム最小化(a)。
- ※Clerkフロー無効化・モック化の提案は本プロジェクトでは**地雷（認証コード触るな制約）に抵触**するため司令塔が却下。

### 司令塔の統合所見
- 3者に共通: **「SWキャッシュ/リロードループ(B)の排除テスト」と「テレメトリ(Sentry)の即時有効化」はどの仮説が真でも無駄にならない**。
- 症状「スケルトンのままロードが終わらない」(F1) は、リロードループ説単体では説明しづらい（リロードなら画面が消えるはず）。**「レンダは生きているがクエリ/エフェクトが暴走している」系（C/E/F）と「リロードループ」(B) の複合または択一**。スクショの静止画では区別不能 → 動画/機械観測が必要。
- 今日(7/4)も6コミットをデプロイしており「デプロイ直後」条件は常に成立してしまっている点に注意（相関と因果の分離が必要）。

## 3. 地雷マップ（過去に壊した箇所・捨てた案・触るな）

1. **ゲスト `/` の tRPC Provider defer境界は触るな**。defer中に `useQuery` を呼ぶと `enabled:false` でも throw する。境界内で呼ぶなら `useTrpcReady()` ゲート必須（過去に本番OOM障害 `4c97daf`→修正 `ad50f8e`）。
2. **認証フロー（`hooks/use-auth.ts` / `server/_core/oauth.ts` / `server/twitter-routes.ts` / Clerk Satellite）は手動E2E計画なしに触るな**（refactor-instructions 確定事項3とも整合）。会議の「Clerkモック化」案はこの理由で却下済み。
3. **`?romiLiteHome=1` キルスイッチと `LIVE_PRESENCE_RADAR_ENABLED` フラグは恒久設計。削除するな**（bisect と緊急止血の手段）。
4. **捨てた案**: 静的LCPシェル（PageSpeed計画で撤回）、`c92ad7b` の revert（直らなかった＝revertで直らない既存バグだった）、地図SVGの「回転停止」「パス間引き」（地図は静止していると実測済み・見当違い）。
5. **`e2e:auth-save` は過去2回「保存成功に見えて空」だった**（偽陽性バグ、`fbfd366` 修正済み）。完了確認は `.auth/auth-state.json` が数KB以上あること。
6. **PowerShell 経由のファイル操作は日本語で化ける**（.gitignore が実際に壊れていた）。検証は実挙動テストで。
7. **`.env.vercel.tmp` が過去コミット `05a2fd6` に残っている**（機密の可能性、未確認）。履歴掃除は別タスク。
8. **タブ切替では expo-router Tabs は画面をアンマウントしない**（web は display:none 保持）→ unmount cleanup は発火しない。「cleanupしたから安全」は成立しない。
9. 「こちらでは再現しない」の最有力説明は **SWキャッシュ残留（端末履歴依存）** と **認証状態の差**。開発環境の正常動作を根拠に安全と断定するな。
10. 検証済みレシピ: prodビルド＋遅延注入サーバー＋headless Playwright で「ロード速度依存のレース」を再現できる（ゲストOOMで実績）。
