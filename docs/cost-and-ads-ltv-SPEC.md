## 0. 全体を貫く一つの思想

このアプリの成長エンジンは「チェックインという中毒的行動」であり、その行動は同時に **(a) DBの行** と **(b) 広告のインプレッション在庫** を生む。つまり、

> **チェックイン1回あたりの限界コストを限りなくゼロに近づけ（Part 1）、チェックイン1回あたりの期待収益をわずかでもプラスにする（Part 2）。そうすれば「いくら地点登録されても気にならない」どころか「されるほど嬉しい」構造になる。**

両パートの判断基準はすべてこの1行に還元する。

# Part 1: コスパ運用設計

## 1.0 まず現実のスケール感を確定する（会議Cの試算への反論）

会議Cの「年間18億行・360TB」試算は **二重に誤っている** ので棄却する。

1. **このアプリは常時トラッキングではなく明示的チェックイン型**（`app/(tabs)/checkin.tsx` が行動の起点）。「1日50件×10万ユーザー」は連続記録アプリの数字。現実のヘビーユーザーで1日10〜30件、DAU 1,000人・平均10件/日でも **年間365万行**。
2. 行サイズの実測見積り: `locations` の1行は id(4)+userId(4)+h3R8(約16)+real×5(20)+municipality/prefecture/address のテキスト(平均50〜120)+timestamp×2(16)+visibility(8)+タプルヘッダ(23)+インデックス3本分 ≒ **250〜400バイト/行（インデックス込み）**。
   - 100万行 ≒ 0.3〜0.4GB
   - 年間365万行 ≒ **1.2〜1.5GB/年**
   - Railway のストレージ課金は $0.15〜0.25/GB・月クラス（現行プランで要確認）。つまり **想定成長の10倍でも月数百円**。

**結論: 今のスケールでは「コスト削減の実装」より「成長速度の監視と、引くべきレバーの事前準備」に投資すべき。** 会議C発散役の「ストレージ絶対量でなく成長速度の微分を見よ」という指摘だけが本質的に正しく、これをPart 1の中心に据える。

## 1.1 アーキテクチャ方針: 「削らない。行単価を下げ、レバーを段階的に引く」

コンセプトの核（正確な lat/lng/accuracyM を永続保存し、後でたどれる）は**一切妥協しない**。代わりに「テーブルサイズの閾値ごとに引くレバーの梯子」を定義し、閾値到達をアラートで検知する。

### レバーの梯子（閾値はすべて `pg_total_relation_size('locations')` 基準）

| 段階 | 閾値 | 引くレバー | 優先度 |
|------|------|-----------|--------|
| L0 | 〜2GB（現在） | 何もしない。監視だけ入れる | P0 |
| L1 | 2GB超 or 週次成長率がプラン残容量を12ヶ月で食う速度 | 複合インデックス最適化 + Hot/Cold読み分け | P1 |
| L2 | 10GB超 | `recordedAt` の B-tree→BRIN 置換、地名テキストの正規化 | P2 |
| L3 | 50GB超 | 月次ネイティブパーティショニング | P2（手順書のみ用意） |

### L1: インデックス最適化（会議C批判役の指摘を採用）

現状のインデックスは `h3R8` / `userId` / `recordedAt` の単独B-tree3本（`drizzle/schema/encounter.ts:58-60`）。主要クエリパターンは「あるユーザーの足あとを時系列で」なので:

- **追加**: `locations_userId_recordedAt_idx` on (`userId`, `recordedAt` DESC)
- **削除**: 単独の `locations_userId_idx`（複合で完全にカバーされる）
- **維持**: `h3R8`（すれ違いマッチング用）、`recordedAt` 単独（sweepの直近ユーザー取得・全体系クエリ用。L2でBRIN化）

インデックス2本→2本の入れ替えなので書き込みコスト増ゼロ、読み取りは改善。`drizzle/migrations/` に通常のマイグレーションとして追加（`pnpm db:push` ではなくマイグレーションファイルで。既存運用に従う）。

### L1: Hot/Cold読み分け（会議C発散役の案3を縮小採用）

「思い出をたどる」（地図リプレイ・足あと便）だけが生の全行を必要とする。一覧・統計・県別パネルは集計で足りる:

- 既存の `visitedAreas` テーブルがすでに「市区町村×初訪問」のサマリとして存在する。**新テーブルを増やす前に、一覧系UIのクエリが `locations` 全走査ではなく `visitedAreas`/`LIMIT` 付きクエリになっているかの監査を先に行う**（実装者タスク: `modules/encounter/db/queries.ts` の各クエリに EXPLAIN をかけ、行数無制限スキャンを洗い出す）。
- マテリアライズドビューは**現段階では導入しない**。REFRESH の運用（誰がいつ回すか）が sweep 頻度問題を再生産するため。必要になったら L2 で `user_daily_summary`（userId, date, 件数, 代表セル）を通常テーブル+チェックイン時インクリメンタル更新で作る。

### L2: BRIN と地名正規化

- `recordedAt` は serial id と物理的にほぼ相関する（追記オンリー）ため、BRIN が理想的に効く。B-tree比で **サイズ1/100以下**。10GB到達時に置換。
- 1行の最大の脂肪は `address`/`municipality` のテキスト。同じ場所への再訪・近隣チェックインで高頻度に重複するため、`place_labels`(id, latGrid, lngGrid, municipality, prefecture, address) への正規化で行あたり50〜100バイト削れる。**ただし JOIN が全読み取り経路に入る変更なので、実測で必要になるまで絶対にやらない。**

### L3: パーティショニング（手順書だけ用意）

月次 RANGE パーティション（`recordedAt`）。Drizzle はパーティション DDL を管理できないため、`drizzle/migrations/` に手書き SQL マイグレーションとして入れる手順を `docs/` に残す。**トリガー（50GB）は現実的に数年先**なので、実装はしない。手順書があれば将来の実装者が迷わない、それで十分。

## 1.2 コスト監視・早期検知（`api-cost-alert.ts` パターンの横展開）

X APIコスト監視の既存パターン（月次アラートキー + `notifyOwner` + `COST_ALERT_WEBHOOK_URL`）はそのまま流用できる。**新規ファイル `server/db-growth-alert.ts`** を追加:

### 収集する指標（発散役の「微分を見よ」を具体化）

sweep 実行時（後述の新頻度で）に以下を取得し、軽量スナップショットテーブル `db_stats_snapshots`(id, capturedAt, locationsBytes, locationsRows, totalDbBytes) に1行 INSERT:

```sql
SELECT
  pg_total_relation_size('locations') AS locations_bytes,
  (SELECT reltuples::bigint FROM pg_class WHERE relname='locations') AS approx_rows,
  pg_database_size(current_database()) AS total_bytes;
```

`reltuples` は推定値で COUNT(*) 全走査を避ける（監視自体がコストになる愚を避ける）。1日1行なら年間365行、監視テーブルの肥大化は問題にならない。

### アラート条件（2軸）

1. **絶対量**: `totalDbBytes` がプラン容量の 70% で警告、90% で危険（`api-cost-alert.ts` の 80%/100% 二段構えを踏襲）
2. **微分**: 直近7日間の増分 × 52 が「プラン残容量」を超える場合に「このペースだとNヶ月で満杯」警告。式: `remainingBytes / (weeklyDelta / 7)` = 残日数。90日を切ったら通知。

通知は `notifyOwner` + Webhook の既存2経路をそのまま使う。アラートキーは `db_growth_{YYYY-MM}` で月1回に抑制（既存パターン踏襲）。

## 1.3 GitHub Actions sweep の見直し（P0・最も費用対効果が高い）

`api/sweep.ts` の現在の実処理を精査した結果:

1. `deleteExpiredLocations` — **常に0件**（永続保存方針でコメントにも明記済み）。無意味。
2. `SELECT 1` keepalive — **Railway の有料 PostgreSQL はアイドルでスリープしない**（Neon/Supabase Free と違う）。keepalive は前提が崩れており不要。
3. `getRecentLocationUserIds` — 件数をログに出すだけで、マッチング回収は未実装。15分ごとに走らせる理由がない。

つまり **15分毎 sweep は「3つとも実質不要な処理」を1日96回、月2,880回実行している**。プライベートリポジトリなら GHA 無料枠 2,000分/月を単独で食い潰しうる（1回≒1分課金）。

### 変更案（P0）

- **cron を `*/15 * * * *` → `0 3 * * *`（1日1回、JST正午≒UTC 3時）に変更。**
- `api/sweep.ts` の役割を再定義: 「keepalive」ではなく **「日次ヘルスチェック + DB成長スナップショット + アラート」** に。
  - `deleteExpiredLocations` 呼び出しを削除（互換の建前ももう不要。コメントごと整理）
  - `SELECT 1` は接続確認として残してよい（害はない）
  - 1.2 の `db-growth-alert` をここから呼ぶ
- 取りこぼしマッチング回収を将来実装する場合も、タイムシフトマッチング（過去30日）というプロダクト仕様上、**日次で十分**。リアルタイム性が必要なのはチェックイン時のオンデマンドマッチングであり、それは既にAPI側で行われている。

効果: GHA 消費 2,880分/月 → 約30分/月（96分の1）。Vercel Functions 呼び出しも同率減。

## 1.5 却下した案と理由

| 案 | 却下理由 |
|----|---------|
| TimescaleDB Hypertable | Railway のマネージド Postgres で拡張が使える保証がなく、使うならカスタムイメージ運用になる。移行リスクと運用複雑性が、数年先まで来ないスケール問題への対処として過剰。ネイティブパーティショニング（L3）で同等の実益を得られる |
| 座標のデルタエンコーディング | 「正確な座標をそのまま読める」単純さを壊し、全読み取り経路に復元ロジックが入る。削減幅は行あたり十数バイトで、テキスト列の脂肪より小さい。バグったら思い出データが壊れる、リスクとリターンが逆転している |
| 48h削除・丸め保存への回帰 | コンセプトの核に反する。議論の対象外 |
| マテリアライズドビュー即導入 | REFRESH の運用主体・頻度問題が sweep 問題を再生産する。必要性が実測されてから |
| sweep 完全停止 | 日次ヘルスチェック＋成長監視の「乗り物」として1日1回は残す価値がある。ゼロにすると監視の置き場所を別に作る羽目になる |

# Part 2: 広告・LTV戦略設計

## 2.0 戦略の前提: このアプリで「広告」が成立する条件

批判役の「広告がUIUXを一切邪魔しないのは不可能」は正しい。だから目標を言い換える: **「行動ループの経路上には置かず、行動の余韻（reward phase）にだけ置く」**。チェックインの習慣ループは 引き金（場所に来た）→ 行動（チェックイン）→ 報酬（マッチ・切手・記録が増えた）。広告が引き金と行動を邪魔すれば習慣が死に、LTVの分母（継続日数）が消える。報酬の後ろに静かに置く分には、ユーザーは「読み飛ばせる」。

もう一つの前提: **制約3・6により外部広告SDK（AdMob等）は使えない**。したがってアドネットワークの自動配信収益は当面存在しない。これを正直に認めた上で、SDKなしで成立する収益源を3層で設計する:

1. **アフィリエイトカード**（旅行・宿・飲食系アフィリエイトリンク。SDK不要、ただのリンク）— 最初の収益
2. **直販スポンサー枠**（聖地巡礼スポット・地元店舗・イベント主催者への「協賛の思い出」枠販売）— 中期の本命
3. **プレミアム（広告非表示を含む）** — 広告と対になる恒常収益

## 2.1 核となるフォーマット: 自社配信ネイティブカード「協賛の足あと」

会議D発散役の案1（Trail Ads）を骨格として採用し、以下のように具体化する。

### フォーマット仕様

- **見た目**: 既存の足あとカード/切手UIと同じレイアウト文法（`theme/tokens/` 準拠）に、左上へ **「協賛」ラベルを常時明示**（ステマ規制対応・2023年10月施行の景表法告示に適合。広告であることの明示は法的必須）。
- **中身**: 画像1枚 + 見出し + 一言 + 遷移ボタン（外部リンク or クーポン表示）。**静止画のみ。点滅・ループ・自動スライド禁止**（制約4）。
- **配信**: `server/routers/ads.ts`（tRPC）が `sponsor_cards` テーブルから配信。ターゲティングは **都道府県/市区町村の粗い粒度のみ**（`locations.prefecture`/`municipality` を既に持っている）。生座標・H3セルをターゲティングに使わない — 批判役のプライバシー懸念への回答であり、「粗い地域×文脈」で十分効く。
- **外部スクリプトゼロ**: 画像も自社ホスティング（`public/` or 既存の画像配信経路）。SWキャッシュ問題（制約5）・バンドルサイズ問題（制約6）・CSPに一切触れない。

### 配置マップ — 入れる場所と絶対に入れない場所

| 場所 | 判定 | 理由 |
|------|------|------|
| **チェックイン完了画面の結果表示より下**（スクロールしないと見えない位置に静的配置） | ✅ 第1枠 | 報酬の余韻フェーズ。行動は完了済みで邪魔しようがない。「チェックイン後5秒バナー」（批判役案）は行動経路に被るので**この形に修正して採用** |
| **図鑑（zukan）の一覧フィード、8枚以上ごとに1枚** | ✅ 第2枠 | 回遊・振り返りフェーズ。切手帳メタファーに「協賛の切手」として自然に混ざる。1画面に同時に2枚見えない密度を保証 |
| **マイページの統計の下** | ✅ 第3枠（P1） | 低頻度・低感情負荷 |
| チェックイン操作中（ボタン・確認・位置取得中） | 🚫 絶対禁止 | 行動経路。ここを汚すと習慣が死ぬ |
| 地図上のピン・マップオーバーレイ | 🚫 絶対禁止 | DESIGN.md「正確な場所が見える地図」に偽情報を混ぜることになる。広告ピンは思い出データへの信頼を毀損する |
| レーダー画面（`(tabs)/index`） | 🚫 絶対禁止 | 「会いたい君がいる現在地」のブランド核体験 |
| モーダル/インタースティシャル/起動時 | 🚫 絶対禁止 | 侵入度最大。離脱direct要因 |
| プッシュ通知広告 | 🚫 絶対禁止 | 通知チャネルはマッチ・すれ違い専用。一度でも広告を流すと通知許可自体が剥がれる |
| `/u/{slug}` シェア着地ページ | ⏸ 当面入れない（P2で再検討） | ゲスト経路は defer 地雷（制約1）の巣であり、かつ「足あと便」SPEC の初訪問体験を汚すリスク > 収益。入れる場合も足あと便リプレイの**再生完了後の静的領域のみ**、defer構造無変更が条件 |

### 「足あと便」SPECとの整合コメント（制約8）

図鑑・タイムラインへの「協賛の思い出」カードは、足あと便リプレイ（`docs/share-landing-uiux-SPEC.md`）の世界観 — 手紙・旅の再生 — と**同じカード文法を使えば競合しない**。ただしリプレイの再生中演出に広告を挟むことは将来も禁止事項としてここに明記する。リプレイは「純粋な記憶」の体験であり、2.3 のプレミアム境界線の論拠でもある。

## 2.2 リワード型・スポンサー解放（会議D案2の縮小採用、P2）

SDKなしでは動画リワードは不可能。代わりに: **スポンサーカードの詳細（クーポン/店舗紹介）を開くと「特別な足あとマーカー」を1つ解放**という自社完結の交換にする。強制視聴ゼロ・完全オプトイン。直販スポンサーが1件でも取れてから実装する（在庫ゼロでガワだけ作らない）。

## 2.3 プレミアムとの境界線

`lib/premium-features.ts` の既存5機能はテンプレート残骸（チャレンジ作成・コラボ等）を含むため、**このアプリの実態に合わせて再定義する**:

| 無料（永久に無料を約束） | プレミアム |
|--------------------------|-----------|
| チェックイン・正確な座標の永続保存 | **広告非表示（「広告なしの純粋な記憶」）** |
| 地図で足あとをたどる | 高度統計ダッシュボード |
| すれ違いマッチ・図鑑・シェア | データエクスポート（GPX/GeoJSON） |
| | 足あとマーカーのカスタマイズ |

**鉄則: 記録と再訪（コンセプトの核）は絶対に課金壁の後ろに置かない。** それは成長エンジンであり、行を増やすことはPart 1により「怖くない」のだから、無料で無限にやらせるのが正しい。会議D案3「プライバシー境界課金」のブランド選択UIは過剰設計として棄却するが、「無料=文脈広告あり/有料=広告なしの純粋な記憶」という境界線の言語化だけを採用した。

**追記（ユーザー方針の確認事項）: すれ違いマッチングは一方向スタンプのみとし、アプリ内でのDM・直接連絡機能は本SPECの範囲に含まない。ユーザー間交流はXへの委譲を維持し、広告・プレミアムいずれの施策もこの境界を変更しない。**

## 2.4 LTV最大化: 短期収益と継続のバランスルール（定量）

LTV = ARPU × 継続期間。このアプリの ARPU 上限は当面低い（自社配信のみ）ため、**継続期間を毀損する広告は数学的に必ず損**。よって:

1. **新規ユーザー保護期間**: 登録後7日間 or 累計10チェックインまで広告ゼロ。習慣形成が先、収益は後。
2. **頻度キャップ**: 1セッション最大1枚（チェックイン完了枠）+ 図鑑回遊で最大2枚。**1日合計3インプレッション上限**。カウントはクライアント側 + サーバー側配信制御の二重。
3. **キルスイッチ**: 配信のON/OFF・枠別ON/OFF・キャップ値をDBのremote config（`sponsor_config` 1行テーブル）で持つ。デプロイなしで止められること（CDNチャンクキャッシュ地雷を踏まえ、挙動制御をバンドルに埋めない）。
4. **撤退基準**: 広告ON群のD7継続率が相対5%以上悪化したら該当枠OFF。計測は既存の分析基盤の範囲で行い、専用A/B基盤は作らない（枠ごとの段階ロールアウトで代替）。
5. **在庫ゼロ時の挙動**: スポンサー/アフィリエイトカードが無い場合、**枠自体を消す**（「広告募集中」のプレースホルダーは絶対に出さない — 汎用SaaS感の極み、DESIGN.md違反）。

## 2.5 技術実装方針（制約との整合）

- **新規スキーマ**: `drizzle/schema/ads.ts` — `sponsor_cards`(id, title, body, imageUrl, linkUrl, prefecture?, municipality?, weight, startsAt, endsAt, active) / `sponsor_config`(単一行: enabled, slotFlags, dailyCap) / `ad_stats_daily`(cardId, date, impressions, clicks)。
  - **Part 1との整合**: インプレッションを行単位で記録しない。日次カウンタへの UPSERT（`INSERT ... ON CONFLICT DO UPDATE SET impressions = impressions + 1`）にし、広告ログが新たな肥大化源になることを構造的に防ぐ。
- **新規ルーター**: `server/routers/ads.ts`（`server/routers/index.ts` への登録必須 — ディレクティブ1）。`getCards`(slot, prefecture) と `track`(cardId, event)。
- **新規コンポーネント**: `components/molecules/sponsor-card.tsx`。コード分割するなら **`lazy()` + Suspense のみ**（制約2: `import()` チャンク直接描画は禁止）。ただしカード1枚は軽量なので、まずは分割せず静的 import で入れてバンドル影響を実測する方が安全。
- **ゲスト経路**: P0/P1 の配置は全てログイン後画面（checkin/zukan/mypage）のみ。`guest-web-providers.tsx`・`TabPrefetchProvider` には**一切触れない**（制約1）。
- **外部ライブラリ追加: ゼロ**（制約3）。アフィリエイトも通常の `<a>`/Linking 遷移で完結。
- **アニメーション: なし**（制約4）。フェードインすら不要、最初から静的に置く。
- **SW/CDN**: 外部スクリプトなし・配信内容はAPIレスポンスなので SW キャッシュ戦略（`lib/service-worker.ts`）は無変更（制約5）。

## 2.7 却下した案と理由

| 案 | 却下理由 |
|----|---------|
| AdMob/GAM等の広告SDK導入 | 制約3・6に正面衝突。PWA/モバイルWebでの動作前提が崩れ、バンドル985KB問題を悪化させ、SW地雷（制約5）の新たな踏み場になる。自社配信で置換可能 |
| チェックイン後5秒の半透明バナー（批判役案） | 「5秒」「半透明オーバーレイ」が結果確認という報酬体験の上に被さる。下部静的カードに修正して吸収 |
| プッシュ通知広告（fast役案） | 通知チャネルの信頼を燃やす。マッチ通知の開封率こそが継続の生命線 |
| 動画リワード広告 | SDKなしでは実装不能。スポンサー解放（2.2）で代替 |
| 地図上の「近隣スポット」広告ピン（批判役案の一部） | DESIGN.md「正確な場所が見える地図」に広告物件を混ぜると、地図＝真実というアプリの信頼基盤が壊れる。地図外のカードなら可 |
| プライバシー境界課金のブランド選択UI（発散役案3） | 広告主ゼロの段階でユーザーに「どのブランドを表示するか選ばせる」UIは完全な過剰設計。「有料=広告なし」の境界線だけ採用 |
| 生座標/H3セル粒度の広告ターゲティング | プライバシー・法規制リスク（批判役指摘は妥当）。都道府県/市区町村粒度で実用十分 |

# 共通: 整合性の総括と制約チェックリスト

## 両パートの整合性（一言で）

**Part 1 が「行が増えるほど困る」構造を潰し、Part 2 が「行動が増えるほど収益在庫が増える」構造を作る。だから両者は矛盾しないどころか、同じ一つの命題 — チェックインの限界コスト≒0・限界収益>0 — の両面である。** 広告ログさえ日次集計主義（2.5）で Part 1 の思想に従わせてあり、思想の漏れはない。
