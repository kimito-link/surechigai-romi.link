## 1. 横断原則（全画面共通）

### 1.1 体験の背骨: 「記録 → 確認 → 収集 → 巡礼」ループ

このアプリの1日は次のループで回る。各画面はループの1工程を担当し、**必ず次の工程への出口を1つ持つ**。

```
Check-in（残す）→ Map（あとで行ける精度で確認）→ Radar/Home（すれ違いが届く）
→ Zukan（集まっていく）→ X シェア → 他人の Map（聖地巡礼）→ その場で Check-in（ループ再入）
```

現状の弱点はこのループの「接続部」にある。チェックイン成功後に地図へ行く動機づけが弱く、図鑑は履歴の羅列で収集感がなく、巡礼者が他人の軌跡を見ても「そこで自分も記録する」導線がない。本書の改善はほぼすべてこの接続部の強化である。

### 1.2 パフォーマンス予算（置き換え原則）

- **無限アニメーション（`withRepeat(-1)`）総数: 認証ホームで9本を上限**（現状値を天井にする）。新しいモーションを1本入れたければ、既存を1本消すか有限回アニメに落とす。
- 新規の常時 DOM/SVG 要素は**1画面あたり純増 +50 要素まで**。超える提案には遅延マウント or 仮想化を必須で併記。
- すべての `withRepeat` は `useEffect` の cleanup で `cancelAnimation` すること（前科の直接原因）。設計書内の各モーションに cleanup 責務を明記した。
- `document.visibilitychange` / `AppState` でタブ非表示時に全モーション pause（既存実装があれば流用）。
- reduced-motion 時: パルス・スイープ・グローは静止画に置換。値の変化はフェードなしの即時反映で良い。

### 1.3 コピー原則（差分だけ）

`DESIGN.md` の禁止語・推奨トーンに加え、本書で使う新規コピーは以下に統一する。

| 用途 | コピー |
|------|--------|
| チェックイン成功 | `足あとを残しました` |
| 精度が良い | `±12m — あとで戻って来られる精度です` |
| 精度が悪い | `±350m — 空の見える場所だと精度が上がります` |
| 未開封すれ違い | `新しいすれちがいが {n} 通` |
| 図鑑の初訪問 | `{市区町村名} に、最初の足あと` |
| 巡礼モード | `この足あとをたどる` |
| 記録一時停止中 | `足あとを一時停止中 — {時刻}に再開` |

## 2. Check-in（最優先画面）

### 2.1 結論

チェックインを「押す→確定を見る→残った実感を得る→地図かXへ抜ける」の**4拍子・合計2タップ**に固定する。gpt-oss-120b の「スナックバー自動保存」は採用しない（誤保存の取り消し UI が常時必要になり、"正確な場所を意図して残す" というコンセプトと相性が悪い）。代わりに**保存前プレビュー→明示保存**を1画面で完結させる。

### 2.2 フロー（確定版）

```
[現在地を記録する] タップ
  ├─ 未許可 → 文脈1文付き許可要求（既存決定。変更しない）
  └─ 許可済み → 測位中状態（ボタン自体がレーダーパルスに変化。新規アニメではなく既存 checkin-cta-button のモーション流用）
       ↓ getCurrentPosition 成功
[保存前プレビューカード]（同一画面内、モーダルにしない）
  ・ミニ地図（既存タイル）に ピン + 精度円
  ・「±{n}m」+ 精度コピー（§1.3）+ 逆ジオコーディング済み市区町村
  ・accuracyM > 100 のとき「もう一度測る」セカンダリボタンを表示（保存はブロックしない）
  ・[この場所に足あとを残す]（primary, 高さ48px）
       ↓ タップ（2タップ目）
[成功状態]
  ・ピンフォーカスモーション1回（有限、withRepeat 不使用）
  ・「足あとを残しました」+ 記録時刻 + 市区町村
  ・出口2つ: [地図で見る]（primary → /map、該当ピンにフォーカスして開く） / [Xでシェア]（secondary）
  ・スナックバー「取り消す」を5秒表示（保存後の undo。自動保存の undo ではない）
```

### 2.3 UX 根拠

- 保存前プレビューは「正確な座標を永続保存する」ことへの**同意の可視化**でもある。何が保存されるかを保存前に見せるのは、プライバシー自衛をユーザーに委ねる方針（移動用アカウント運用）と整合する。
- 精度が悪いときに保存をブロックしない理由: 地下・屋内では ±300m でも「その日そこに居た」記録価値がある。ブロックすると離脱する。改善手段（再測位）だけ提示する。
- 成功→`/map` 直行は §1.2 ループの最重要接続。「残した足あとが地図上に正確に在る」ことを保存直後に見せるのが、このアプリ最大の Aha モーメント。

### 2.4 リスク・地雷との関係

- **watchPosition は使わない**。`getCurrentPosition`（timeout 10s、`maximumAge: 15s`）の単発呼び出しに限定。会議指摘の clearWatch 漏れリスクを構造的に消す。「もう一度測る」も単発再呼び出し。
- 保存ボタンは押下後即 `disabled` + 測位中スピナー化（二重タップ防止。debounce ではなく状態遷移で防ぐ）。
- プレビューのミニ地図はタイル画像 1〜4 枚 + SVG ピン/円のみ。MapLibre 等の新規ロードはしない（地雷4）。
- 位置許可 UI には一切触れない（地雷1）。

### 2.5 実装コンポーネント（`components/checkin/` 配下、優先度付き）

| 優先度 | コンポーネント | 内容 |
|--------|---------------|------|
| **P0** | `checkin-preview-card.tsx`（新規） | 保存前プレビュー。ミニタイル地図 + ピン + 精度円 + 精度ラベル + 市区町村。props: `lat/lng/accuracyM/municipality/onSave/onRetry` |
| **P0** | `checkin-success-panel.tsx`（新規） | 成功状態。ピンフォーカス1回モーション（`withTiming` のみ、cleanup 不要な有限アニメ）+ 出口2ボタン + undo スナックバー |

## 3. Map（軌跡 — 価値の証明画面）

### 3.1 結論

会議で「精度円と周辺文脈が競合する」と指摘された問題は、**ズームレベルで表現を切り替える**ことで解決する（両方を常時描くから競合する。どちらを主役にするかをズームに語らせる）。加えて「足あとカード」ボトムシートを導入し、Apple「探す」的な「地図＋詳細シート」の二層構造にする。

### 3.2 ズーム別情報設計（本書の中核提案・その1）

| ズーム | 主役 | 精度の表現 | 周辺文脈 |
|--------|------|-----------|----------|
| z ≥ 16（街区） | **精度円**（塗り: primary 12% opacity、線: primary 1px）+ ピン | 円そのもの | タイルの建物・道路がそのまま文脈。ラベル追加不要 |
| z 13–15（市区） | **ピン** | 円は描かず、選択中ピンのラベルに `±{n}m` をテキスト表示 | タイルの駅名・幹線道路 |
| z ≤ 12（都市〜全国） | **クラスタ点**（数字入り円形バッジ。円形は「ピン」として意味があるので契約上許可） | 非表示 | 市区町村名はカード側で担保 |

原則: **精度円は「その場に行ける」ズームでしか意味を持たない**。引きの地図で円を描くのは視覚ノイズであり、会議指摘の競合はこの原則で構造的に消える。

### 3.3 足あとカード（ボトムシート）

ピン/クラスタのタップで下からシートが出る（画面下 40%、上スワイプで 70%）。ネストカード禁止に従い、シート自体が唯一のカード。

```
┌─────────────────────────────┐
│ 東京都 渋谷区                    ← title 16px, textPrimary
│ 6月28日(土) 14:32 · 5日前        ← body 14px, textSecondary
│ 35.6595, 139.7005  ±12m         ← monospace 12px（DESIGN.md: 座標は等幅）
│ 「初めての現場。空が広かった」      ← ひとこと（あれば）
│ [ここへ向かう] [Xでシェア] [⋯]    ← ⋯内: 公開/非公開切替・削除（既存 hooks 流用）
└─────────────────────────────┘
```

- `[ここへ向かう]` は既存 `navigate-to-place-button.tsx` を流用。
- 削除・公開切替は既存 `useTrailLocationActions` / `DeleteTrailConfirmModal` をそのまま使う。
- シート表示中、該当ピンに**ピンフォーカス**モーション（許可モチーフ）を1回。有限アニメ、cleanup 明記不要だが `withRepeat` は使わない。

### 3.4 軌跡線と時間フィルタ

- 時系列で足あとを結ぶ**軌跡ポリライン**（primary 40% opacity、2px）。「移動の足あと」というコンセプトを地図上で唯一直接表現する要素。点が2個以上ある日のみ描画。
- ヘッダー直下に時間フィルタチップ（`filter-tabs.tsx` 流用）: `今日 / 今週 / 今月 / すべて`。`myTrail` クエリの limit/期間パラメータに接続。
- 選択期間の要約を地図左上にオーバーレイ1行: `12 足あと · 3 市区町村`（地図上オーバーレイのみ白文字許可の契約に従い、背景に primary 85% のピル）。

### 3.5 巡礼モード（コンセプト直結の独自提案）

`/u/[slug]`（既存の公開プロフィールルート）から他人の公開足あとを開いたとき、Map を**巡礼モード**で表示する。

- ピンを時系列の**番号付きピン**（①②③…）にし、軌跡線を表示。
- 足あとカードに `[この足あとをたどる]` → 端末の地図アプリへ経路を渡す（`navigate-to-place-button` の対象違い）。
- 自分の現在地が対象ピンから 100m 以内に入った状態でチェックインすると、成功画面に `聖地に足あとを重ねました` の1行を追加（判定は端末内で完結。サーバー保存は通常のチェックインと同一。新規スキーマ不要）。
- **これが「推しの軌跡をファンが聖地巡礼できる」の実装形**であり、巡礼者を Check-in ループに引き込む接続部（§1.2）。

### 3.6 リスク・地雷との関係

- ポリライン・クラスタは既存 `LazyWebTrailMap`（`components/organisms/web-trail-map/` 系）内の描画追加であり、認証済み chunk 内で完結。ゲスト境界に触れない（地雷3）。
- クラスタリングにより z≤12 での DOM 要素数はむしろ**減る**（120 ピン → 十数クラスタ）。§1.2 予算内。クラスタ計算は新規ライブラリを入れず、グリッドビンニングの純関数 `lib/cluster-locations.ts` で実装（120点なら十分）。
- ボトムシートは reanimated の有限アニメ（`withSpring` 1回）。無限アニメ0本追加。

| **P0** | `footprint-sheet.tsx`（新規, `components/map/`） | 足あとカード。§3.3 の構成。既存 actions hooks 接続 |
| **P0** | ズーム別精度表現 | `web-trail-map` 内の描画分岐。§3.2 の表 |
| **P0** | `?focus={locationId}` 対応 | Check-in 成功からの着地。該当ピンへ pan + フォーカスモーション + シート自動オープン |

## 4. Zukan（図鑑 — 達成感・希少性の設計。本書の中核提案・その2）

### 4.1 結論

会議で「単なる履歴一覧」と指摘された弱点に対し、**「切手帳」メタファー**で再設計する。ランキングやバッジの一般ゲーミフィケーション（凡庸案）ではなく、「正確な場所を残す」コンセプトから達成感を導出する: **達成の単位は市区町村、希少性の単位は"最初の一枚"と"すれ違い圏内での被り"** とする。

### 4.2 画面構成（上から順）

```
1. コンプリートヘッダー
   「{n} / 47 都道府県」 大数字（hero 30px, primary）+ 細プログレスバー（8px, 角丸4px）
   「{m} 市区町村 · すれ違い {k} 人」 body 1行
2. 日本地図（既存 japan-block-map 流用、塗り分けを4状態に拡張）
   未踏 = background / 足あとあり = surfaceEmphasis / すれ違いあり = primary /
   今週新規 = accent の小ドットを県上に1つ（静的。パルスさせない — 地雷2予算節約）
3. 切手帳グリッド（市区町村カード 2列）
4. すれ違い相手の出身地（既存セクション維持）
```

### 4.3 市区町村カード＝「切手」仕様

角丸8px・非ネストの単層カード。装飾ではなく**データで差をつける**。

```
┌──────────────────┐
│ 渋谷区        東京都  │ ← title 16px / meta 12px textMuted
│ 初 2026.5.12         │ ← 「最初の一枚」の日付。monospace
│ 足あと 8 · すれ違い 3  │
│ ● 一枚目             │ ← 希少性タグ（該当時のみ、accent 12px）
└──────────────────┘
```

**希少性タグの定義（実装可能な3種、上位1つだけ表示）:**

| タグ | 条件 | データソース |
|------|------|-------------|
| `一枚目` | 自分のすれ違い相手グラフの中で、この市区町村に足あとを持つのが自分だけ | `visitedAreas` を encounters の相手集合で絞る集計 |
| `{n}人目` | 同グラフ内で n 番目の到達（n≤5 のみ表示） | 同上 + firstVisitedAt 順位 |
| `再訪 {n}回` | 同一市区町村への訪問回数 5/10/25 の節目 | 既存 `municipalitySummary` 相当の集計 |

- 「全ユーザー中の希少性」ではなく「**自分のすれ違い圏内での希少性**」にするのが要点。全体ランキングは廃墟化するが、すれ違い相手という小さな社会圏なら「あの人より先にこの街を踏んだ」が実感を持つ。DS のすれ違い通信の楽しさ（狭い世界での被り/独占）の再現でもある。
- この集計はリクエスト毎に走らせず、**サーバー側で日次バッチ or 図鑑クエリ時の集計 + 短TTLキャッシュ**とする（tRPC `zukan.myAreas` の応答に `rarity` フィールドを追加する形。クエリコストは実装時に EXPLAIN で確認 — 検証ゲート）。

### 4.4 ソート・フィルタ（凡庸案の採用形）

チップ1列のみ: `新しい順 / 古い順 / 足あと数順`。既存 `filter-tabs.tsx` 流用。都道府県での絞り込みは既存ルート `zukan/[prefecture]` があるため、日本地図の県タップで遷移させる（新規フィルタ UI を作らない）。

### 4.5 カードタップ → Map 接続

切手カードのタップで `/map?municipality={key}` に遷移し、その市区町村の足あとにズームする。図鑑を「見る場所」から「たどり直す入口」に変える（§1.2 ループ接続）。

### 4.6 リスク・地雷との関係

- 新規アニメ 0 本。「今週新規」ドットも静的。日本地図の塗り分けは既存 SVG のクラス変更のみで要素数純増なし。
- カードにミニ地図サムネイルを載せる案は**不採用**（カード数×タイル画像でメモリ・転送とも地雷2/4に抵触。タップで Map に飛べば足りる）。
- 希少性集計は認証済み API の変更であり、ゲスト境界（地雷3）に触れない。

### 4.7 実装コンポーネント（優先度付き）

| 優先度 | コンポーネント | 内容 |
|--------|---------------|------|
| **P0** | `zukan-complete-header.tsx`（新規, `components/zukan/`） | 47都道府県プログレス + 集計1行 |

## 5. Radar/Home（感情の入口 — 3秒で状態が読める画面）

### 5.1 結論

OOM 前科画面のため、**要素を増やす方向の改善を全部禁じ、情報の序列を組み替える方向で改善する**。この画面が3秒で答えるべき質問は3つ: (1) 新しいすれ違いは届いているか、(2) 今日、足あとを残したか、(3) 次に押すのはどこか。gpt-oss-120b の「感情スロット 2列3行」は**不採用**（タップ先が6個に増え、チェックイン導線が割れる。1タップ再設計 SPEC の思想と逆行）。

### 5.2 画面構成（上から順）

```
1. TabScreenHeader（既存のまま）
2. ステータスライン（新規・1行）
   「今日の足あと: 渋谷区 14:32」 or 「今日はまだ足あとがありません」
   ＋ 記録一時停止中なら「足あとを一時停止中」ピル（accent 枠、MyPage §7 と連動）
3. レーダー（既存 japan-radar-map / radar 系を維持。sweep リング1枚・アニメ9本の現状構成を変えない）
   中央に自分のアバター（既存）
4. 封筒レール（新規・横1列）
   未開封 encounters を封筒カード（既存 envelope-card.tsx）で横スクロール表示。
   「新しいすれちがいが {n} 通」見出し。0件時は「すれちがいを待っています」1行 + レールごと非表示
5. プライマリ CTA: [現在地を記録する] → /checkin（既存 checkin-cta-button 流用）
```

### 5.3 UX 根拠

- 封筒は DS すれ違い通信の情緒の核であり、既存資産（`envelope-card` / `encounter-open-modal` / `envelope-pulse`）が揃っている。ホームの一等地に置くことで「開けに戻ってくる」リテンション動線になる。
- ステータスラインは「今日まだ」を可視化して1日1チェックインの習慣を作る。テキスト1行なのでコストゼロ。

### 5.4 リスク・地雷との関係（この画面は特に厳格に）

- 封筒レールの表示上限 **5通**（それ以上は「ほか {n} 通」テキスト）。`FlatList` horizontal + `windowSize` 小さめで、DOM 純増を上限 ~40 要素に抑える（§1.2 予算内）。
- `envelope-pulse` を封筒に使う場合、**パルスは先頭の1通だけ**に限定し、`useEffect` cleanup で `cancelAnimation` を必ず書く（前科の直接原因への明示的対策）。これで無限アニメは 9→10 本になるため、**レーダー側の既存9本から1本を有限化して枠を空ける**（置き換え原則）。どれを落とすかは実装時に `post-authenticated-screen.tsx` 内の `withRepeat` を列挙して決める。
- **Canvas + rAF によるレーダー描画（gpt-oss 案）は P2 の検証付き仮説**として扱う。移行条件: (a) 実機（メモリ2GB級 Android）で現行 SVG 構成の Chrome DevTools メモリプロファイルを取り、(b) Canvas 試作で同条件比較し、(c) ARIA ラベル（`role="img"` + 状態のテキスト代替）と画面回転リサイズを実装できた場合のみ。**効果が実測で確認できなければやらない**。地雷2の cleanup 削減（残り9本の未完分）のほうが先。
- ゲスト側 `PostGuestScreen` と認証ゲートには一切触れない（地雷3）。全変更は `post-authenticated-screen.tsx` 内。

### 5.5 実装コンポーネント（優先度付き）

| 優先度 | コンポーネント | 内容 |
|--------|---------------|------|
| **P0** | `home-status-line.tsx`（新規, `components/post/`） | 今日の足あと1行 + 一時停止ピル。`myTrail` の最新1件から導出（追加クエリ不要なら既存データ流用） |
| **P0** | `envelope-rail.tsx`(新規) | 未開封封筒の横レール（上限5 + パルスは先頭1通のみ + cleanup 必須）。タップで既存 `encounter-open-modal` |
| **P1** | 残存 `withRepeat` の cleanup 完遂 | 地雷2の未完対策（9本の cleanup 監査）。UI 変更ではないが本画面の前提整備として P1 に置く |
| **P2** | レーダー Canvas 化 | §5.4 の検証ゲート3条件を満たした場合のみ。満たさなければ**恒久見送りで良い** |

## 6. MyPage / Settings（静かで高密度・誤操作しない）

### 6.1 結論

演出は足さない。1038行の現行画面を**5グループの情報アーキテクチャ**に再編し、危険操作を物理的に隔離する。既存の `mypage-action-list.tsx` / `my-signal-summary.tsx` / `confirm-modal.tsx` を流用。

### 6.2 情報アーキテクチャ（上から順）

| グループ | 項目 |
|---------|------|
| 1. プロフィール | アバター・X アカウント表示・**[あなたの公開ページを見る]**（`/u/[slug]` プレビュー導線。自分がどう巡礼されるかを自分で確認できる — 巡礼機能の信頼の土台） |
| 2. 記録の設定 | **足あとの一時停止**（下記）／新規足あとの公開デフォルト（公開/非公開）／**自宅マスク**（P2、下記） |
| 3. すれ違いの安全 | ブロック一覧／通報履歴（既存機能の導線整理） |
| 4. データ | 足あとエクスポート（P2）／**すべての足あとを削除**（最下部・後述） |
| 5. アプリ情報 | バージョン・利用規約・special-thanks 導線 |

### 6.3 足あとの一時停止（`userSettings.locationPausedUntil` の UI 化）

- 3択チップ: `1時間 / 今日中 / 手動で再開するまで`。選択で即時反映、確認モーダル不要（可逆なので）。
- 停止中はグループ2の先頭に `足あとを一時停止中 — {時刻}に再開` を accent 枠ピルで常時表示し、**Radar/Home のステータスライン（§5.2）にも同じピルを出す**。停止したまま忘れて「記録されてなかった」となるのがこの機能最大の事故なので、二画面で可視化する。
- コピーに「位置情報は保存されません」系の禁止語を使わない。「足あとを**一時停止**」で統一。

### 6.4 危険操作の隔離

- 「すべての足あとを削除」はページ最下部、ボーダー区切りの下、通常リストと8px×3以上離す。文字色 textPrimary（赤の多用はしない — パレット外）。タップで既存 `confirm-modal` を使い、**削除対象の件数を明示**（「{n} 件の足あとを完全に削除します。元に戻せません」）＋ ボタンラベルは「削除する」でなく「{n} 件を削除する」。
- ログアウトは既存 `logout-confirm-modal` のまま。削除系と同じ視覚グループに置かない。

### 6.5 自宅マスク（`homeMaskCell` — P2）

- 地図で自宅周辺セル（H3 r8 ≒ 460m）を1つ選ぶと、**そのセル内の新規足あとが自動で非公開デフォルト**になる（保存自体はされる。マッチングには使われる。公開だけ抑制）。
- 設定 UI は Map の流用（セル選択モード）。「移動用アカウント運用に委ねる」方針を補完する自衛オプションであり、方針転換ではないことを実装コメントに明記。

## 8. 全体実装順序（結論）

**フェーズ1 — コアループを閉じる（最優先）**
1. Check-in P0（プレビューカード・成功パネル・状態機械化）— アプリの心臓。ここが2タップで気持ちよく回らない限り他の全画面に素材（足あと）が供給されない
2. Map P0（足あとシート・ズーム別精度表現・`?focus=` 着地）— Check-in 成功の受け皿。「あとで行ける精度」の価値証明
3. Radar/Home P0（ステータスライン・封筒レール）— ループの再入口。※着手前に P1 の `withRepeat` cleanup 監査を先に済ませること

**フェーズ2 — 収集と信頼**
4. Zukan P0/P1（コンプリートヘッダー・切手カード・地図塗り分け・Map 接続）
5. MyPage P0/P1（IA 再編・一時停止 UI・危険操作隔離）
6. Map P1（ポリライン・時間フィルタ）

**フェーズ3 — 差別化（各 P2、検証ゲート通過が条件）**
7. 巡礼モード（§3.5）— コンセプト「推しの聖地巡礼」の実装形。公開プロフィールの流通が始まってから
8. Zukan 希少性タグ（§4.3）
9. 自宅マスク・エクスポート・（実測が正当化する場合のみ）レーダー Canvas 化

**不採用と理由の記録（次の担当者の地雷マップ用）:** 感情スロット6分割（導線分裂）／チェックイン自動保存スナックバー（意図なき永続保存とコンセプト不整合）／緯度経度手入力化（価値と矛盾）／図鑑カードのミニ地図サムネ（メモリ・転送コスト）／全ユーザー横断ランキング（廃墟化・すれ違い圏内希少性で代替）／レーダー廃止して一覧化（感情の入口を失う）／画像・音声メモ（ストレージ/モデレーション負荷）。
