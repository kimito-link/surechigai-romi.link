## A. 水面（結章「湖に、月が、逆さに浮かんでゐる」）の現状と診断

### 設計と実装（済み）
- 設計SPEC: **`docs/lp-mizu-redesign-SPEC.md`（Fable完成版・552行）**。3段構え（会議→Fable→実装）の成果。
  height field 2バッファ水面 / 月ディスクをオフスクリーンに円で描き高さ場でサンプリング変位 /
  「月の道」/ 凪の報酬 / 雫と月の琴の音 / QAフック `window.__mizu` / 実行番地つき。
- 実装: 別モデルが Phase 1-4 を実装・デプロイ済み（`ceee01fe9`・`app.js?v=28`）。
  該当コード: `public/lp/app.js` の mizu IIFE（`renderWater`/`stepWave`/`poke`/`waterMoon`相当/
  `shizuku`/`tsukigoto`/`window.__mizu`）、`public/lp/index.html` の結章 canvas（1009-1012行）+ CSS（536-560行）。

### 司令塔Claudeが検証で掴んだ事実（重要・次チャットはここから）
- **前回の「巨大な月光の柱」は完全に解消**。canvas単体のピクセルを `toDataURL` で書き出すと、
  濃紺の夜空・金の月・星・富士の稜線・**水面に映る丸い逆さ月**が美しく描けている（SPEC通り）。
  → 描画ロジック自体は成功している。
- **QAフックは機能**: `window.__mizu.poke(0.68,0.4,1)` で `energy` が上がり波が伝播（0.035等）。
- **しかし実際の画面表示（DOM合成スクショ）では画面がベージュに化ける**。canvasバッファは
  濃紺（サンプル `[11,23,46]`）なのに、DOM合成表示で背景（和紙色/tsuki-mizu.png?）が透ける。
  `.tmp-mizu-handoff.png`（本チャットで撮影）が「あまりよくない」実例。
- **canvas の `getBoundingClientRect().top` が -468〜-1572**（画面外上方）。sticky でも
  一時的に position:fixed に変えても同様。祖先に transform/filter/perspective/will-change は無い
  （fixed を壊す典型原因は不在）。結章 section の高さは約1088px と短く、`.mizu-holder{position:sticky;
  top:0; height:0}` の粘着範囲が足りずcanvasがsection下端で剥がれる、という仮説はあるが、
  fixed化しても直らなかった（むしろ画面全体がベージュに悪化）ので**真因は未確定**。

### ⚠️ 次チャットへの最重要警告
- **司令塔Claudeのヘッドレス Playwright 検証は信頼性を失った**。canvasバッファは濃紺なのに
  DOM合成表示だけベージュ = **ヘッドレス（deviceScaleFactor）固有のアーティファクトの疑いが濃厚**。
  **実機での見え方が未確認**。まず**ユーザーの実機**（またはヘッドフルのブラウザ）で
  `https://surechigai.kimito.link/lp/` の結章を実際に見て、「濃紺の水面に丸い逆さ月が見えるのか、
  ベージュに化けるのか」を確定させること。ヘッドレスのスクショだけで判断して修正を重ねると
  悪化する（司令塔が実際にやらかした。fixed化で全画面ベージュにした→revert済み）。
- Playwright を使うなら **`reducedMotion:"no-preference"` 必須**（渡さないと水面が起動しない）。
  そして `toDataURL`（canvasバッファ）と 通常screenshot（DOM合成）の**両方**を撮り、差があれば
  それはDOM側/CSS側/検証環境側の問題であって描画ロジックの問題ではない、と切り分ける。

### 水面の直し方針（実機で症状を確定させてから）
- もし実機でもベージュに化けるなら: 原因は canvas の位置（section内 sticky が短い結章で機能しない）
  か、canvas より前面/背後の要素の重なり（`#photoStage` opacity, `.veil`, `.ph::before`）。
  `body.mizu-active #photoStage{opacity:0}` は効いている（確認済み）。次に `.veil`・和紙地・
  `photo-bg` 背景が canvas の下から透けていないかを実機の devtools で層ごとに確認。
- SPEC死守条件「position:fixed 全画面Canvasを二度と使わない」の**真意は「常時fixedで座標がずれ
  見切れるのを防ぐ」**。もし実機で「section内sticky/absoluteでは短い結章に対応できない」と確定
  したら、`data-mizu`可視中だけ JS で fixed 化する案は真意に反しないが、司令塔の試行では
  ベージュ悪化したので**慎重に**（fixed化そのものより、その時のcanvasクリア/描画順の問題かも）。

## B. 四季情景のSD画像 全面作り直し（ユーザーがSDを回す）

### 現状の画像
- `public/lp/img/` に四季の情景写真（SD生成）が数十点。`public/lp/img/_prompts.txt` に生成時の
  プロンプトメモがある（例: かき氷=「melting red strawberry syrup」だが実画像は青いブルーハワイ
  だった＝文と絵の不一致で削除済み、という前例）。
- ユーザー評価: **「道の描写に統一感がない」→ 完全に変える**。全画面背景で使われボケが目立つものが対象。

### ⚠️ 重要: SD は Claude(このエージェント)からは実行できない
- GPU・SDモデルウェイト・画像生成APIが手元に無い。**SDを回すのはユーザーのローカル環境**
  （AUTOMATIC1111 / ComfyUI 等）。「起動して」はユーザー側の操作。
- 次チャットのClaudeができるのは: ①統一感のある**プロンプト設計**（四季を貫く画風・構図・
  色調・「道」のモチーフの一貫性）を作る ②`_prompts.txt` を刷新 ③生成後の画像を LP に差し込み
  ④Playwrightで見え方確認。**画像生成そのものはユーザーが実行**。

### 道の統一感を出すプロンプト設計の方針（次チャットが詰める）
- 現LPの世界観: 芭蕉「月日は百代の過客」/ 和紙・墨・明朝・旧字体 / 四季道中記。
- 「道」が全季を貫くモチーフ（雪道・桜並木の道・夏の細道・秋の細道・夜富士への道）。
  現状バラバラなので、**同一の画風トークン**（例: 統一した lighting / palette / 画角 / 墨絵or写実
  のどちらかに寄せる / aspect ratio）を全プロンプトに固定し、季節差分だけ変える設計にする。
- ユーザーと「墨絵寄り」か「写実写真寄り」か、SAKURA(墨絵桜KV)を超える方向性を最初に合意する。
