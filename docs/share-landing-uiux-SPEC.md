## 0. 現状の問題の言語化

現状のページ（`app/u/[slug].tsx`）は「他人の管理画面を覗いている」体験になっている。
アバター＋地図＋履歴リストという**情報の羅列**であり、シェアした本人が込めた
「いま、ここにいるよ」という時間性・物語性が死んでいる。具体的には:

1. 着地した瞬間に全ピンが一括表示され、**「どう動いたか」が読み取れない**（順序はPolylineに畳み込まれて不可視）
2. 訪問者がすることが何もない（見る→閉じる）。**参加の余地ゼロ**
3. `retry: false` により、モバイル回線の一瞬の不調で「共有リンクが見つかりません」という**偽のエラー**が出る（シェア導線として致命的）
4. ヘッダー「軌跡」という無機質な語。誰の何のページかがヘッダーからは分からない

裏取りで判明した重要事実（会議素材の取捨選択に直結）:
- サーバーは**既に強力なプライバシー機構を持っている**: デフォルトは500mグリッド丸め
  （`shareLocationPrecise`がtrueの人だけ正確な座標）、`homeMaskCell`除外、地点ごとの
  `visibility`、`trailVisibility`、`paused`。会議Bの懸念の大半は既に実装済みで、
  足りないのは「それを訪問者に見せる」ことだけ。
- `PrecisionTileMap`は**既に軌跡Polyline（react-native-svg）と履歴ドットを描画済み**。
  つまり「番号ピン＋軌跡線」の情報量は現状と同じで、リプレイ演出は新たな情報を一切追加しない。
- `lib/accessibility.ts`にprefers-reduced-motion判定、`lib/share.ts`等にX intent導線が既存。
  新規ライブラリはゼロで足りる。

## 1. 核となるアイデア: 「足あと便」— 届いた手紙を開くと、旅が再生され、現在地に追いつく

### コンセプト

Xで受け取ったリンク = **その人から届いた1通の「足あと便」**。開くと:

- **幕1（着地〜約1秒）**: 絵葉書風のヒーローカード。「◯◯さんから、足あと便」。消印風に最新地点の地名と相対時刻
- **幕2（〜約8秒、スキップ可・1回きり）**: 地図上で足あとが**古い順に1つずつ点灯**し、カメラが追いかけてパンする。各停車地で地名チップがポップ。最後に現在地マーカー（アバター）が着地し、精度円が1回だけパルス。コピー **「いま、ここ。」**
- **幕3（追いついた後）**: 地図が通常のインタラクティブ表示に戻り、(a) タップで「ここで、すれ違えたかも」波紋（ゴーストすれ違い）、(b) 「この足あと便をXでシェア」、(c) ログインCTA が**この順で**現れる

### なぜこの判断か（根拠）

| 判断 | 根拠 |
|---|---|
| リプレイ（時間軸の再生）を核にする | 会議A案2「スクロール連動ジャーニー」の情緒を、RN Webで確実に動く技術（setTimeoutチェーン＋既存の`customCenter`パン）に置換したもの。「会いたい君がいる**現在地**」というブランドコピーの「現在地」に**追いつく**という体験は、静的な地図では絶対に出せない差別化。かつ既存データだけで実装でき、新情報を露出しない |
| 絵葉書/消印メタファー | 会議A案1と、既実装のZukan切手帳メタファー（uiux-brushup Phase1-3）とブランド連続性がある。DSすれ違い通信の「手紙が届く」原体験とも一致 |
| ゴーストすれ違いを幕3に採用 | 会議A案3のうち「保存しない・ログイン不要・位置情報許可不要」の部分だけ抽出。会議Bの「訪問者の大半は聖地に行けない」への回答: **行かなくてもその場で擬似すれ違いできる**。制約5（位置情報許可タイミング不変）にも抵触しない |
| 実ジオフェンス巡礼チェックインはP2に降格 | 会議Bの指摘どおり大多数の訪問者に無関係で、100m判定・順序管理のコストが高い。「たどる」価値の90%はリプレイ＋ゴーストで先に回収できる |
| 番号付きピンは不採用 | 順序は「再生アニメ＋履歴リスト」で伝わる。恒久的な番号ピンは「1番=自宅近辺」のような読み解きを**静止画スクショで**誘発する（会議Bの生活パターン逆算懸念）。リプレイなら動画的にしか順序が見えず、かつ再生対象を「直近の旅」に限定できる |

### 2.2 リプレイの仕様（`use-trail-replay.ts`）

**入力**: `locations`（APIは新しい順で返す。hookが`recordedAt`昇順にソート）

**再生対象の決定（「直近の旅」セグメンテーション）**:
1. 最新地点から過去へ遡り、隣接地点の時刻差が**6時間以内**なら同じ旅とみなして連結
2. 上限**20地点**。1地点しかなければ旅は再生せず、現在地マーカーの着地演出のみ
3. 再生対象外の古い足あとは、リプレイ中も**最初から薄いドット＋Polylineで背景表示**（既存描画のまま）。「歴史は見える、ドラマは直近」

**状態機械**: `idle → playing(step: n) → arrived`（再生し直しで `playing(0)` へ戻る。無限ループなし＝制約4遵守）

**駆動**: `setTimeout`チェーン（1ステップ約650ms、20地点でも最大13秒→地点数に応じ450〜650msに可変させ**8秒以内**を目標）。各ステップで:
- `revealedPoints = 旅[0..n]` を`PrecisionTileMap`の`locations`に渡す（配列先頭=最新の既存契約を維持するため、渡す際は逆順化）
- `customCenter = 旅[n]`（既存propsだけでカメラ追従が成立）
- `zoom`: 再生中は`fitCenterZoom`（既存export）で旅全体が収まるズーム、`arrived`で16へ
- 地名チップ: 直前ステップの`municipality`をトースト風に1個だけ表示（opacity 0→1→0、Animated一発）

**arrived演出**: 精度円を1回パルス（scale 1→1.4→1, opacity減衰、Animated.sequence 1回）＋「いま、ここ。」ラベル表示。以後静止。

**スキップ/再生**:
- 再生中: 右下に「スキップ」ピル → 即`arrived`
- `arrived`後: 「▶ 旅を再生しなおす」ボタン（トリガー起点なので何度でも可）
- **reduced-motion**: `lib/accessibility.ts`の既存判定がtrueなら初期状態を`arrived`にする（再生ボタンは残す）

**自動再生の発火条件**: `trailQuery.data`到達 && 旅が2地点以上 && 初回のみ（`useRef`でガード）。タイルの読み込み待ちで白地図に線だけ走るのを避けるため、開始を600ms遅延。

### 2.3 ゴーストすれ違い（`ghost-crossing-layer.tsx`）

- `arrived`後のみ有効。地図タップで:
  1. タップ座標に波紋1回（1.2秒で消滅、Animated一発）
  2. 小さなゴーストマーカー（半透明の足あとアイコン）＋吹き出し「ここで、すれ違えたかも」を約2.5秒表示して消える
- **保存しない・APIを呼ばない・位置情報を使わない**（P0時点）。同時表示は最大3個、超過分は最古を即消す
- 実装: `PrecisionTileMap`の上に`pointerEvents`制御した絶対配置Viewを重ねる。座標→pixel変換は既存の`projectPoint`/`pixelToLatLng`をimport（`precision-tile-map.tsx`からexport済み）
- 注意: `PrecisionTileMap`の既存`onPointPress`（ピンタップ→`FootprintSheet`）と競合するため、ゴースト層はピンのhitSlopを避ける — 実装はゴースト層を**下**、ピンPressableを**上**のz順にし、地図素タップだけ拾う

### 2.4 CTA・シェア導線

- **`VisitorCtaBar`**（未ログインのみ）: `arrived`後に下から1回フェードイン。コピー: 見出し**「あなたの足あとも、誰かの『会いたい』になる」**＋既存`KimitoLoginCta`（returnTo=`/u/{slug}`維持）。現行の地図上部`InlineLoginPrompt`は**廃止**（物語が始まる前に売り込む形になっていたため）
- **`ShareThisTrailButton`**（全員に表示）: X intentで `text=「{名前}さんの足あと便をたどった📍{最新エリア名}」 url={現URL}`。**訪問者自身がリシェア主体になれる**のが「もう一度シェアしたくなる」の第2エンジン（第1はオーナー自身がページの見栄えに満足すること）
- ログイン済み訪問者: CtaBar非表示、代わりに履歴リスト下に「自分の足あと便をつくる →」（`/checkin`へのリンク）

### 2.5 エラー処理の修正（地味だがP0）

`app/u/[slug].tsx`:
```ts
trpc.ogp.getTrailBySlug.useQuery(
  { slug, limit: 120 },
  {
    enabled: /^[A-Za-z0-9]{1,16}$/.test(slug),   // ← 変更しない（制約2）
    retry: (count, err) =>
      err?.data?.code !== "NOT_FOUND" && count < 2, // NOT_FOUNDは即確定、他は2回リトライ
  },
)
```
`isError`分岐を2種に分割:
- `NOT_FOUND`: 現行文言のまま
- それ以外: 「読み込みに失敗しました。電波の良いところでもう一度」＋**「再試行」ボタン**（`trailQuery.refetch()`）。「都道府県別一覧へ」は残す

## 3. データ構造・APIの変更点

**P0では変更なし。** 根拠:

- 順序: `recordedAt`が全地点に含まれており、クライアントで昇順ソート・旅セグメンテーションが完結する
- 精度情報: レスポンスに`precise`（本人のshareLocationPrecise設定）が既に含まれ、PrivacyBadgeの表示にそのまま使える
- `featureShareLocationFirst`による「シェア地点を先頭へ」の並べ替えは、地図の初期センター用として維持（リプレイhookは自前ソートするので影響なし）

**P2で追加するもの**（採用する場合のみ）:

```
ogp.recordGhostCrossing (publicProcedure, mutation)
  input:  { slug, h3R8 }        // タップ地点をH3 res8に丸めてから送る（生座標を送らない）
  効果:   share_ghost_counts へ日次集計upsert（個人特定情報なし・IPはレート制限にのみ使用）
  返却:   { totalCount }
表示:    ヒーローに「この足あと便をたどった人 12人」
```
匿名カウンタは社会的証明としてリシェアを加速するが、スパム対策（slugごと・IPごとのレート制限）が必要なのでP2。

## 4. プライバシー方針（会議Bへの回答・確定事項）

**方針: 「新たな露出を増やさず、既にある保護を可視化する」**

1. **座標の粒度は現行サーバー実装を変更しない**。デフォルト500mグリッド（`latGrid`/`lngGrid`）、`shareLocationPrecise`をオンにした本人のみ正確な座標。±200mランダムシフト案（会議B改善案1）は**不採用** — 既にグリッド丸めという決定的で説明可能な機構があり、ランダムシフトは「後で行ける」というプロダクト転換方針と矛盾する
2. **`PrivacyBadge`（新規・ヒーロー内）で粒度を訪問者に明示**: `precise=false`→「📍 おおよその場所（約500m単位）で表示中」/ `precise=true`→「📍 本人の設定により正確な位置を表示中」。懸念を隠すのではなく信頼表示に転換する
3. **番号付きピンは恒久表示しない**（§1の根拠表参照）。順序の露出は「1回きりの再生」と「既存の履歴リスト」に限定
4. **リプレイ対象を直近の旅（≤20地点・6時間ギャップ区切り）に限定**。全履歴120点の移動順序をアニメで強調しない
5. `homeMaskCell`・地点ごと`visibility`・`trailVisibility`・`paused`は既存のまま全経路で尊重（コード変更なし＝リグレッションなし）
6. 法的論点（個人情報保護法の第三者提供）は会議の一次情報を鵜呑みにしない。ただし方向性対応として、`paused`時の文言・本人による地点単位の非公開機能が既にあることをREADME/利用案内で明文化することを実装外タスクとして推奨

## 7. 却下した案と理由

| 案（出典） | 却下理由 |
|---|---|
| 番号付きピン恒久表示（旧巡礼モード案） | スクショ1枚で生活パターンが読める（会議B）。順序の価値はリプレイ＋履歴リストで代替可能 |
| スクロール連動の地図パン（会議A案2） | scroll-timelineはRN Webで使えず、ScrollView＋IntersectionObserverはプラットフォーム差異のデバッグコストが高い。「時間をたどる」情緒はリプレイで同等以上に達成 |
| 地図タイルへの紙質感CSSフィルタ（会議A案1の一部） | DESIGN.mdの禁止事項「正確な場所が見えない地図UI」に抵触。紙メタファーは**地図の外側**（ヒーローの絵葉書枠・消印）に限定して採用 |
| デバイス傾き/マウス視差（会議A案3の一部) | 会議内でも実装難度高と評価。効果に対しコスト過大、モーション酔いリスク |
| 周辺POI表示（会議A・llama案） | 外部API依存（コスト・制約3抵触の恐れ）かつ「その人の足あと」という主役をぼかす。凡庸 |
| クエスト型・実座標を隠すランダム数カ所提示(会議B改善案3) | 「正確な場所を残し後でたどれる」というプロダクト転換方針の核と真っ向から矛盾 |
| 座標±200mランダムシフト（会議B改善案1） | 既存の500mグリッド丸めが上位互換。二重の丸めは「後で行ける」価値を壊す |
| バーチャル巡礼（実座標に基づかない代替・llama-4-scout案） | 同上。実在の場所であることがこのアプリの存在理由 |
| 訪問者の現在地とリアルタイム距離比較（会議A批判役の「自分ごと化」案） | 訪問者への位置情報許可リクエストが新たに発生し、制約5の精神（許可タイミングの設計は確定済み)に抵触。ゴーストすれ違いが許可不要の代替 |
| 公開デフォルトオフ化（会議B改善案2） | `trailVisibility`/地点別`visibility`/`paused`という段階的コントロールが既に存在。シェアリンクは本人が能動的に発行するものであり、デフォルト非公開化はシェア機能自体の否定になる |

## 8. 制約遵守チェックリスト

- 制約1: 新規重量UIは`LazyShareTrailExperience`として`lazy()`+`Suspense`登録のみ。`import()`直描画なし
- 制約2: `enabled`条件・`hasClerkSessionInStorage`・provider初期化は不変更
- 制約3: 新規ライブラリ0（react-native-svg/Animated/既存タイル地図/既存ヘルパーのみ）
- 制約4: 無限アニメ0本、全てトリガー起点1回きり
- 制約5: 位置情報許可フロー不変更（ゴーストすれ違いは許可不要設計）
- 制約6: `VisitorCtaBar`のfixedは`as unknown as object`キャスト踏襲
- 制約7: 実装フェーズで`CDN_CACHE_EPOCH`+1＋マーカー文字列での配信確認を手順化
