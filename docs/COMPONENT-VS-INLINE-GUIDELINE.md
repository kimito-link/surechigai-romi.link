# コンポーネント化 vs 直書きの判断ガイドライン

**方針**: 「直書き0」ではなく、**“変わるもの”だけをコンポーネント化（または設定化）**し、**“変わらない骨格”は直書きOK**にする。

---

## 1. 直書きしていいもの（＝骨格）

- 画面の**レイアウト構造**（Grid/Stackの並び）
- ページ固有の**ストーリー**（そのページだけの流れ）
- 1回しか出ない**超固有UI**（再利用予定なし）

## 2. コンポーネント化すべきもの（＝変わる・増える）

- 同じUIが**2回以上**出る（Button/Card/Modal/Tableなど）
- バリデーション、エラーハンドリング、ローディングなどの**共通挙動**
- 「表示ルール」があるもの（IP表示の強調、ステータス表示、通知）

## 3. コンポーネントより“設定化”が向いてるもの

- 文言（コピー）やラベル → **constants/copy** または i18n
- APIのURL、タイムアウト、閾値 → **config**
- 画面の表示条件（ON/OFF）→ **feature flags**

## 4. 判断ルール（迷ったらこれ）

| 条件 | 対応 |
|------|------|
| 同じものが2回出る | コンポーネント化 |
| 表示や仕様が変わりやすい | 設定化 |
| そのページでしか成立しない | 直書きOK |
| 共通の挙動（loading/error/empty） | 共通コンポーネント or 共通hook |

## 5. 本プロジェクト（doin-challenge.com）の現状マッピング

### 現在の構成

| 役割 | パス | 役割 |
|------|------|------|
| UI部品 | `components/ui` | Button, Card, Modal, Input 等 |
| 原子・分子・生物 | `components/atoms`, `molecules`, `organisms` | 階層構造 |
| 業務部品 | `features/*` | イベント、参加、プロフィール等 |
| 状態・通信 | `hooks` | useAuth, useHomeData 等 |
| API | `lib/trpc`, `lib/_core/api` | tRPC, fetch 共通化 |
| 定数 | `constants/copy`, `constants/*` | 文言、閾値、ラベル |
| ページ | `app/` | Expo Router の組み立て |

### 方針との整合性

| 推奨 | 現状 | 評価 |
|------|------|------|
| UI部品 | `components/ui` | ✅ あり |
| 業務部品 | `features/*` + `organisms` | ✅ あり |
| hooks | `hooks/` | ✅ あり |
| API | `lib/` | ✅ あり |
| 定数 | `constants/` | ✅ あり |
| ページ | `app/` | ✅ あり |

## 6. 最小手戻りで改善できる点

### 6.1 直書きのままでもよい（変更不要）

- `app/(tabs)/index.tsx` 等のページレイアウト（Grid/Stackの組み合わせ）
- 1ページでしか使わないモーダルやフォームの流れ

### 6.2 設定化の余地

- 文言がハードコードされている箇所 → `constants/copy/` へ移動
- マジックナンバー（閾値、タイムアウト）→ `constants/` または `shared/const.ts`

### 6.3 コンポーネント化の余地

- **loading/error/empty** の繰り返し → `ScreenLoadingState`, `ScreenErrorState` 等が既にあるか確認し、不足分を追加
- **チャレンジカード** 等の重複UI → `features/events` 配下で統一されているか確認

### 6.4 状態を表すUIの共通化（優先度高）

動員チャレンジは **参加状態（未参加/参加済/主催）** が多いため：

- ステータスバッジ、参加ボタンの有無、空状態
- これらを `StatusBadge`, `ParticipationButton`, `EmptyState` 等の共通コンポーネントにまとめると効く

---

## 7. まとめ

現状の構成は、このガイドラインとおおむね整合している。  
無理に全部コンポーネント化するより、「2回以上出る」「変わリやすい」ところから優先的に共通化するのがおすすめ。
