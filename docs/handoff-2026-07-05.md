## 1. 今日中に解決した技術障害（すべて本番反映済み・裏取り済み）

過去のセッションで「チェックインが失敗する・遅い」「リロードのたびに画面がちらつく」という
報告から着手し、調査の過程で以下5つの独立した障害を発見・修正した。**すべて本番デプロイ済み**。

| # | 障害 | コミット | 詳細 |
|---|---|---|---|
| 1 | アイコンJSの動的import→React19無限再レンダー→全ページ凍結→OOM | `b3337de` | `docs/investigation/auth-home-oom-root-cause.md` |
| 2 | presence.pulseの無限ループ（mutationオブジェクトをuseCallback依存に入れていた） | `9efa72d` | `docs/investigation/surechigai-clerk-token-storm`相当（メモリ参照） |
| 3 | Clerkトークン毎回skipCache取得→429嵐 | `dc3bde9`→`7eddee0`で実働フォールバック復帰 | 同上 |
| 4 | CDN同名別内容チャンク→デプロイしても修正が配信されない | `4780846` | `docs/investigation/surechigai-cdn-chunk-cache-landmine`相当 |
| 5 | チェックイン下部ドックがモバイルタブバーと二段に重なる | `534a9de` | スワイプ時にアドレスバー収縮でズレる |
| 6 | `/api/sweep`がVercel未実装で404し続けていた | `834f82a` | 15分毎スイープが機能していなかった |

**重要な教訓（次モデルも厳守）**:
- アイコンJS等の**動的importでReactコンポーネントを直接描画して常時マウントするパターンは禁止**（`lazy()`+Suspenseは可）。
- **デプロイ確認は`version.json`一致だけでは不十分**。Metroの子チャンクは親チャンクのファイル名に参照先を含めないため、CDN/ブラウザキャッシュに旧チャンクが残り得る。`scripts/bust-entry-cache.cjs`がHTML内の全チャンク参照に`?v=<sha>-<ts>`を付与する対策済みなので、通常デプロイなら問題ないはずだが、疑わしい時はチャンク連鎖をcurlで辿って確認する。
- React Queryのmutation/queryの「結果オブジェクト」をuseCallback/useEffectの依存に入れない（安定参照でないため無限ループの原因になる）。

## 3. Codexに渡す想定の次タスク: 事業設計から出てきた技術対策（最優先）

### 3-1. 公開読み取りAPIへのレート制限追加（最優先・低コスト）

**背景**: Vercelで異常アクセス（過去24h平均Edge Requests約93→直近5分で約7,300）が観測された。
調査の結果、presence.pulse無限ループ（#2、修正済み）が主犯の可能性が高いが、
**副次的に「読み取り系の公開tRPCエンドポイントにレート制限が一切ない」ことが判明した**。
これはバズった際やBot/スクレイパーからの連打に対して無防備。

**対象ファイル**: `api/trpc/[trpc].ts`

現状の仕組み（既存、そのまま流用できる）:
```ts
const expensivePathRules: RateLimitRule[] = [
  { pattern: /^presence\.pulse$/, windowMs: 30_000, max: 1 },
  { pattern: /^encounter\.checkIn$/, windowMs: 15_000, max: 1 },
  { pattern: /^visit\.report$/, windowMs: 20_000, max: 1 },
  { pattern: /^event\.resolveOfflineLocation$/, windowMs: 20_000, max: 1 },
];
```
IPベース、`rateLimitByKey`というMapで管理、10,000件上限で自動GCされる。**新規ライブラリ不要**。

**やること**: 以下のpublicProcedureをこのルールに追加する（書き込みほど厳しい制限は不要、
「1秒間に1回まで」程度の緩い制限で十分。目的はBot連打対策であって通常利用を妨げないこと）:
- `zukan.activePrefectures`（`modules/encounter/api/zukan.ts:28`）— DB集計クエリ
- `zukan.creatorsByPrefecture` — `locations`全期間スキャンで重い（既知）
- `ogp.getTrailBySlug`（`server/routers/ogp.ts:86`）— シェア経由で急増しうる
- `ogp.getShareBySlug`（`server/routers/ogp.ts:60`）

推奨ルール例（要検討・調整可）:
```ts
{ pattern: /^zukan\.activePrefectures$/, windowMs: 5_000, max: 3 },
{ pattern: /^zukan\.creatorsByPrefecture$/, windowMs: 5_000, max: 3 },
{ pattern: /^ogp\.getTrailBySlug$/, windowMs: 5_000, max: 5 },
{ pattern: /^ogp\.getShareBySlug$/, windowMs: 5_000, max: 5 },
```

**検証**: `pnpm check`→`pnpm test`→ローカルで該当エンドポイントを連打して429が返ることを確認→
本番デプロイ後、通常利用（ページ遷移程度の頻度）でレート制限に引っかからないことを確認。

### 3-2. DB接続プールとRailwayプランの整合確認（要判断・実装は保留）

`server/db/connection.ts`の`max: 2`（postgres.js接続プール）は妥当な初期値だが、
Vercel Functionの同時実行数が増えるとRailway側の総接続数上限に達する可能性がある。
**Stop And Ask**: Railwayの現在のプラン・同時接続数上限を確認してから`max`値の調整要否を判断すること。
DBスキーマ変更ではないが、Railway課金プランに影響しうるためユーザー確認必須。

## 4. 事業設計ドラフト（要点、詳細はチャット履歴の直前メッセージ参照）

ユーザーから「事業責任者AI」として20項目の事業設計を依頼された。技術部分は具体的に回答し、
価格・法人展開・特許商標は「たたき台」として提示し断定を避けた。要点:

- **事業定義**: 「移動ログ・コレクションアプリ」というカテゴリ。位置情報アプリと名乗らない。
- **狙うべき層の優先順位**: 1.推し活・配信者クラスタ 2.ドライブ/車中泊層 3.旅行/スタンプラリー層 4.一般層
- **課金モデルの設計思想**: 「見る・繋がる・シェアする」は無料無制限、「貯める機能の高度活用」を課金対象に。
  記録を消させる無料プランは核となる価値提案を壊すので避ける。
- **価格ドラフト**: 無料0円 / ライト300-400円 / プレミアム900-980円 / 法人は案件ごと見積+月額固定。
  **未検証、市場テストを経てから確定すべき**と明記済み。
- **法人向け機能**（キャンペーン管理・スタンプラリー等）は現状DBスキーマに存在しない。新規テーブルが必要。
- **KPI**: 「ユーザー数の増加とAPIリクエスト数の増加が比例しているか」が人気/バグの判別軸。
- 収支の具体的な数字・特許商標の断定的助言は意図的に避けた（未検証の仮定や専門外領域のため）。

## 5. 保留中の別要望（次に着手する場合の入口）

ユーザーから以下2件の機能要望があったが、事業設計を優先するため設計は未着手:

1. **ピンチズーム地図体験**（iPhone写真アプリのような、拡大すると位置精度が上がる体験）。
   - 現状の地図実装（`components/organisms/precision-tile-map.tsx`）は+/-ボタンのみで、
     ピンチジェスチャー・パンドラッグの実装が一切ない。
   - `react-native-gesture-handler`(~2.28.0)・`react-native-reanimated`(~4.1.6)は**既に依存に入っている**ので
     新規ライブラリ追加なしで実装できる。
   - 座標変換の純関数（`lib/map/tile-geo.ts`の`pixelToLatLng`/`latLngToWorldPixel`）は既にあるので、
     ジェスチャー→ズーム値・中心座標の更新ロジックを追加するだけで済む設計。
   - 対象画面はユーザーに確認したところ「Fableで最高の体験を考えて」という回答だったため、
     次回セッションで対象画面（公開共有ページ/自分の軌跡/図鑑/チェックインのいずれか）を
     決めてから設計するとよい。

2. **グルメマップ・思い出機能**（食べた物の写真＋店舗情報を地図に記録）。
   - 現状、`locations`テーブルに写真・グルメ関連のカラムは一切ない（新規テーブルが必要）。
   - ユーザーが送った写真（そば・ラーメン等）からイメージするに、「チェックイン時に写真を添付できる」
     「地図上で写真サムネイルがピンになる」体験を求めている可能性が高い。
   - **DBスキーマ変更を伴う規模の大きい新機能**なので、着手する場合は先に会議ハーネス→Fable設計の
     3段構えプロセスを踏むことを推奨（このプロジェクトでの通常の進め方）。
