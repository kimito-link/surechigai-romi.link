# QA ツールキット・ロードマップ — 「なおってない」ループから抜けるための検証基盤の全体設計

作成: Fable 5（2026-07-04）
関連文書:

- `docs/auth-home-oom-diagnosis-v2.md` — OOM 診断書
- `docs/auth-home-soak-HOWTO.md` — ソークハーネス手順（v2）

## 0. TL;DR

1. これまで直らなかったのは対策の質の問題である前に、**「観測なき修正ループ」**に居たから。
   症状は「なおってない」の1bitでしか返らず、仮説が当たっても外れても学習ゼロのまま修正を積んでいた。
2. ユーザー報告の原文（qa-toolkit-design.md §0）により症状はかなり特定された:
   **初回アクセスの急性クラッシュ**（重い→一瞬見える→白→out of memory）。ただし故障モードは
   複数あり得て（§2）、うち半分は「OOM 対策」では直らない種類のもの。
3. 最も費用対効果が高い一手は新規開発ではなく **(a) `pnpm e2e:auth-save` のログイン完了（人間3分）**と
   **(b) Sentry の DSN 設定（人間5分・無料枠）**。(a) が無いと無人検証は全部ゲスト画面止まり
   （今日まで実際にそうだった）。(b) はコードが統合済みで、エラー時セッションリプレイ
   （＝ユーザー実機の症状の録画）まで設定済みのまま眠っている。
4. 無人検出側は本日2本立てが揃った: **first-load プローブ**（急性・初回マウント）+
   **soak v2**（滞在中・症状非依存: RELOAD_LOOP / BOUNDARY_LOOP / MUTATION_STORM / ERROR_STORM / HEAP_GROWTH）。

## 1. なぜ今のアプローチが繰り返し失敗しているのか（メタ分析）

### 1-1. 現在のループの構造

```
コードレビューで仮説 → 修正 → tsc 0 を確認 → デプロイ → ユーザーに聞く
  → 「なおってない」(1bit) → 次の仮説 → …
```

このループには**測定器が1つもない**。各サイクルで得られる情報は1bitだけで、

- どの症状が（クラッシュ? ちらつき? フリーズ? リロード連発?）
- いつ・どの画面で・どの操作の後に
- 端末のメモリ/コンソールは何を吐いていたか

が一切得られず、**仮説空間が全く狭まらない**。Phase 0/1/2 と対策を積んでも改善報告が無いのは、
対策が外れているからかもしれないし、当たっているが別の故障モードが残っているからかもしれない——
**どちらなのか判別する手段が無い**ことが問題の本体。

### 1-2. 「もっと OOM 対策を積む」ことの限界

- OOM 対策は「症状 = メモリ枯渇によるタブクラッシュ」という仮説が正しい場合にのみ効く。
  症状がリロードループや再レンダ暴走なら、アニメ本数を51→11に減らしても体感は変わらない。
- Phase 1/2 の効果は一度も実機で測定されておらず、「直ったはず」の上に次の対策を積む
  構造になっていた（診断書 v2 自身が指摘）。
- **検証ツール側も同じ罠に落ちていた**: `.auth/auth-state.json` が27バイトの空状態
  （`{"cookies":[],"origins":[]}` = ログイン未完了のまま保存）で、ソークハーネスは一度も
  本物の認証済み画面を計測できていなかった。「検証ツールが検証されていない」の実例。
  → 本日、soak v2 / first-load プローブの両方に「空 auth-state の即エラー検出」を実装済み。

### 1-3. 構造的な根本原因

**認証済みホームを、人間の目でもセンサーでも、誰も継続的に観測していない。**

- Claude は X OAuth を通せない（storageState 方式で解決可能だが、初回の人間ログイン1回が未完了）。
- ユーザーは開発ループの外にいて、リリース後に体感だけを報告する。
- 本番実機で何が起きたかを教えるテレメトリ（Sentry）は無効化されたまま。

直すべきは個々のバグの前に**この観測の空白**。方向は3つで、全部やる:

1. **実ユーザーの実機をセンサーにする** — Sentry + Session Replay + 診断オーバーレイ
2. **無人ハーネスの症状網羅化** — first-load プローブ + soak v2（本日完了）
3. **人間の目の定期化** — ビジュアルリグレッション、テスター配布キット

## 2. 症状の故障モード・マトリクス

ユーザーの症状語（「ちかちか」「白くなる」「out of memory」）に畳み込まれ得る故障モードを列挙し、
検出手段を対応させる。**左列を埋めずに対策を選ぶことが、これまでの失敗の本体。**

| # | 故障モード | 症状の見え方 | OOM対策で直る? | 機械検出 | 担当ツール |
|---|---|---|---|---|---|
| A | **フルリロードのループ**（SWの古いHTML × 新デプロイのchunkハッシュずれ → chunkエラー → `chunk-recover` 自動リロード → また失敗 → …） | 画面全体が周期的に白→再描画（「ちかちか」の字義に最も近い） | ✗ | `page.on("load")` 回数 | soak v2: RELOAD_LOOP |
| B | **ErrorBoundary の remount ループ** | 地図領域が点滅・現れては消える | △ | `[RadarStageBoundary]` console.error 回数 | soak v2: BOUNDARY_LOOP |
| C | **再レンダ/マウントの暴走**（state更新ループ・無限useEffect） | 全体がちらつく・無反応・発熱 | ✗ | MutationObserver childList 変異/秒 | soak v2: MUTATION_STORM |
| D | **初回マウントの急性メモリ枯渇**（SVG 700要素 + クエリ同時発火 + reanimated 初期化の重畳） | 重い→一瞬見える→白→OOM表示 ← **ユーザー報告そのもの** | ✅ | 500ms間隔ヒープ + 白画面ステートマシン + 連続スクショ | first-load プローブ |
| E | **フリーズと復帰の繰り返し**（long task） | カクカク・タップ無反応→急に動く | △ | PerformanceObserver longtask | 両方（参考値） |
| F | **滞在中の漸進的リーク** | 数分使うと重くなり最後に落ちる | ✅ | ヒープ線形回帰 | soak v2: HEAP_GROWTH |
| G | **アニメ/CSS の視覚グリッチ**（opacityパルス衝突・フォントswap・layout shift） | 特定要素の明滅・初期表示の跳ね | ✗ | paintFps の lite 比較・CLS・スクショ比較、最終的に目視/リプレイ | soak v2 参考値 + 短期のVRT |

注記（A について）: `lib/pwa/chunk-recover.ts` の「1回だけリロード」ガードは `load` イベントで
解除されるため、「load 完了後に遅延 chunk の取得が失敗する」状況では**実質リロードループになり得る**。
PWA の SW キャッシュ残留は端末の履歴に依存し、**クリーンな開発環境では原理的に再現しない**。
「こちらでは再現しない・実機では壊れている」の説明として D と並ぶ有力候補。Sentry が入れば
`ChunkLoadError` の実数として即座に判別できる。

## 3. 無人でどこまで検証できるか（境界線の宣言）

### 3-1. 無人で検出できる（本日以降可能）

- 初回マウントの急性クラッシュ・白画面遷移（first-load プローブ）
- タブクラッシュ・ヒープ単調増加・DOMリーク（soak v2）
- リロードループ・remountループ・mutation暴走・エラーレート異常（soak v2）
- 機能スモーク（既存 `pnpm e2e` / `e2e:audit`）
- レイアウト崩れの回帰（短期で VRT 追加後）

### 3-2. 無人では原理的に苦手（実機テレメトリ or 人間が必要)

- 実機固有の描画バグ（GPU/ドライバ依存）、iOS Safari 実機固有問題
- **実機の SW キャッシュ残留状態**（§2-A。その端末の履歴に依存）
- 体感品質（カクつきの「感じ」）、実モバイル回線の挙動

### 3-3. 境界線を動かす3つの構造改革

| 改革 | 何が変わるか | コスト |
|---|---|---|
| **(1) Sentry 有効化** | 3-2 の大半が「実機からの自動報告」に変わる。エラー時セッションリプレイで症状がそのまま録画で届く | 人間5分 + $0（無料枠） |
| **(2) storageState 認証の成立** | 3-1 の全てが「認証済み画面に対して」実行可能になる | 人間3分（1回だけ） |
| **(3) ユーザー実機の remote debugging 手順化** | 「その端末でしか起きない」問題に、USB + `chrome://inspect` で実機 DevTools を直接見る。問題の端末そのものなので BrowserStack より正確 | 人間15分/回・$0 |

## 4. 「サービスとして提供して問題ないレベル」の QA 定義

P0 が欠けている限り「テスターに渡せる状態」ではない。

### P0 — 落ちたことを知れる・症状を特定できる（最低ライン）

| 要素 | 現状 | 手段 |
|---|---|---|
| 本番エラーの自動収集 | ❌ DSN未設定で無効 | Sentry 有効化（`lib/sentry.ts` 実装済み） |
| エラー時の画面録画 | ❌ 同上 | Sentry Replay（`replaysOnErrorSampleRate: 1.0` 設定済み） |
| 急性クラッシュの無人再現 | ✅ 本日 | `pnpm qa:first-load` |
| 滞在中異常の無人検出 | ✅ 本日 | `pnpm soak:auth-home`（v2） |
| リリース前スモーク | ✅ 既存 | `pnpm e2e` / `e2e:audit` |
| 認証済み画面の無人アクセス | ❌ auth-state 空 | `pnpm e2e:auth-save`（人間1回） |

### P1 — 劣化に気づける

| 要素 | 現状 | 手段 |
|---|---|---|
| ビジュアルリグレッション | ❌ | Playwright `toHaveScreenshot`（既存e2e基盤・新規依存ゼロ） |
| パフォーマンス予算の定点観測 | △ 手動 | PageSpeed 資産（`docs/investigation/`）の GHA cron 化 |
| 実機診断値をユーザーが読める | ❌ | `?romiMemDebug=1` オーバーレイ（診断書 v2 Phase 3-3。未実装） |
| SW/デプロイずれの自衛 | △ chunk-recover のみ | リロードループ・ブレーカー（60秒3回で停止→「更新してください」表示 + Sentry 報告） |

### P2 — テスターに渡せる・回せる

| 要素 | 現状 | 手段 |
|---|---|---|
| アプリ内フィードバック導線 | ❌ | 「不具合を報告」→ 診断情報（heap/SW状態/UA/version）自動添付 |
| テスター配布キット | ❌ | 観点チェックリスト + 症状語彙表（§2 の A〜G を非技術者の言葉で）+ 報告テンプレ |
| 実機マトリクス | ❌ | まず無料（ユーザー端末 + 知人の iPhone）。Sentry の UA 統計で不足が見えたときだけ有料検討 |
| CI 常設 | ❌ | e2e + VRT を PR ごとに GHA（無料枠内） |

**「サービス提供レベル」の運用定義**: テスターが症状を報告したとき、**こちらが追加の質問をしなくても**
（Sentry のイベント/リプレイ + 診断オーバーレイのスクショで）症状を再特定できる状態。
テスターの仕事は「見つけること」であって「診断すること」ではない。診断材料の収集はアプリが自動で担う。

## 5. ロードマップ

### 今すぐ（今日〜今週。人間の作業は合計10分未満）

1. **[人間・3分] `pnpm e2e:auth-save` で X ログインを最後まで完了させる**
   完了確認: `wc -c .auth/auth-state.json` が数KB以上あること（27バイトなら未完了）。
   これが無いと first-load / soak の認証済み計測は全部絵に描いた餅。
2. **[人間・5分] Sentry DSN の設定**
   - sentry.io で無料アカウント（Developer プラン）→ React プロジェクト作成 → DSN コピー
   - Vercel 環境変数に `EXPO_PUBLIC_SENTRY_DSN=<DSN>` を追加 → **再デプロイ**（ビルド時埋め込みのため必須）
   - 以後、実機の JS エラー・ChunkLoadError・クラッシュ前エラーが自動収集され、エラー発生
     セッションは画面録画つきで届く（`maskAllText: true` 済みなので文字は出ない）
3. **[Claude・済] 無人検出の2本立て**: first-load プローブ（急性）+ soak v2（滞在中・症状非依存）。
   auth-save 完了後に `pnpm qa:first-load` → `pnpm soak:auth-home --minutes=10` と
   `--lite` の A/B を実行し、§2 マトリクスのどの行が現実かを最初に1回スクリーニングする。

### 短期（1〜2週間）

4. **`?romiMemDebug=1` 診断オーバーレイ**: heap / DOMノード / SW 状態 / version を画面隅に表示。
   ユーザーのスクショ1枚が診断データになる。
5. **リロードループ・ブレーカー**（§2-A の止血 + 検出）: `chunk-recover` に短時間リロード回数の
   上限を付け、超えたら止めて更新案内 + Sentry 報告。
6. **ビジュアルリグレッション最小構成**: ゲストホーム / 認証ホーム(lite) / checkin / zukan の4枚から。
   マスク対象: 時刻・アバター・動的データ。
7. **Sentry の1週間分のデータで §2 マトリクスを事実で埋め直し、対策の優先順位を確定する。**
   （ここが「修正を積んでから祈る」から「測ってから直す」への転換点）

### 中期（1〜2ヶ月）

8. **CI 常設**: PR ごとに e2e スモーク + VRT（GHA）。first-load / soak はデプロイ後 or 夜間 cron。
   認証 storageState の GHA シークレット化が前提。
9. **アプリ内フィードバック導線**（P2）: 診断情報自動添付の「不具合を報告」。
10. **テスター配布キット**（P2）: チェックリスト + 症状語彙表 + 報告テンプレ。ベータテスターは
    X のフォロワーから数名（推し活アプリの性質上リクルートは容易なはず）。
11. **実機マトリクスの要否判断**: Sentry の UA/端末統計で問題が特定端末に偏るなら、まず
    その端末の remote debugging（無料）。それでも足りない時だけ BrowserStack を月単位契約。

## 6. コスト比較（外部サービス）

| 手段 | 費用 | 得られるもの | 判断 |
|---|---|---|---|

## 8. Stop & Ask

- Sentry 無料枠をエラー数が突破した場合（= エラー大量発生の証拠。課金より先に原因対処）
- soak v2 の通常/lite A/B で「lite でも異常」が出た場合（レーダー外が原因。診断書の想定を超える）
- first-load プローブで `CRASH_ON_FIRST_LOAD` が**ゲストでも**出た場合（別系統の問題）
- VRT の期待画像更新が頻発する場合（マスク設計の見直し）
