# Part 1: 本番安定性

## 1.1 根本原因パターン（会議の指摘の評価と、自分の言葉での再整理)

会議E発散役の「境界層における暗黙的な状態同期の欠如」は方向として正しいが、抽象的すぎて対策に落ちない。実際の6件をコードベースで見直すと、**3つの型**に分かれる:

| 型 | 名前 | 該当障害 | 共通構造 |
|---|---|---|---|
| 型1 | **宣言と実体の乖離** | 拡張子なしimport（tsc通過→ESM実行失敗）/ `_journal.json`不整合（ジャーナル≠実DB）/ version.json一致≠配信チャンク一致 | 「宣言側（型・ジャーナル・メタデータ）だけを見てOK判定した。壊れるのは実体側（ランタイム・DB・CDN配信物）」 |
| 型2 | **参照同一性の錯覚** | `import()`直描画→React19無限再レンダー / mutationオブジェクトをuseCallback依存に→無限ループ→429嵐 | 「毎回新しい参照が生成されるものを、安定した同一物だと思い込んだ」 |
| 型3 | **独立バージョンを持つキャッシュ層** | CDN同名別内容チャンク / SW壊れバンドル固着 | 「オリジンを直しても、独立したライフサイクルを持つキャッシュ層は追随しない」 |

**AI並行開発がこれを増幅する理由**（批判役の「体制が悪い」への回答）: 各エージェントは**自分のセッション内で検証可能な層**（tsc、ローカル実行、直近のdiff）しか見ない。型1〜3はすべて「セッションの外側にある状態」（本番DB、CDN、ユーザーのブラウザ、別セッションのpush）で壊れる。つまり体制を変えなくても、**「セッションの外側との突き合わせ」を人の記憶ではなく機械のゲートに置けば増幅は止まる**。これがPart 1・3全体の設計原則。

なお批判役の「責任の所在が曖昧」については、handoff文書とコミット単位の分離が既に機能しており（`e8a8d9e4f`のsweep修正、`9eadea798`のラベル修正はどのセッションが何をしたか追跡できている）、追加の仕組みは不要と判断する。

## 1.2 `_journal.json`不整合の修復手順（最優先・No-Go条件）

### 事実関係（本設計時点で裏取り済み）

- `drizzle/migrations/` にSQLファイルは `0000`〜`0010` の**11本**存在する
- `drizzle/migrations/meta/_journal.json` のエントリは `0000`〜`0004` の**5件のみ**
- `drizzle/migrations/meta/` のスナップショットは `0000`〜`0003` の**4枚のみ**（**`0004_snapshot.json`すら欠落**。0004の時点で既に手作業追加が始まっていた形跡）
- `0005`〜`0010` の内容は `scripts/apply-*.cjs`（`apply-0005-location-visibility.cjs` 等）による**直接ALTER**で本番に反映済み（handoff 11章）
- `pnpm db:push` = `scripts/run-drizzle.cjs push` = **`drizzle-kit generate` → `drizzle-kit migrate`**（`drizzle-kit push`ではない。pushコマンドは`force-push`に隔離済み）

### 壊れ方は「二重」であることに注意

1. **generateが壊れる**: generateは「schema TS ⇔ 最新スナップショット」の差分でSQLを作る。最新スナップショットが`0003`なので、0004以降の全変更を含む**巨大な偽マイグレーション**が生成される（しかも既存の`0005_location_visibility.sql`とファイル番号が衝突しうる）。
2. **migrateが壊れる**: `0005`/`0006`のSQLは`IF NOT EXISTS`なしの生`ADD COLUMN`なので、二重適用で即エラー。

したがって「journalに6行足すだけ」では直らない。**スナップショット・journal・DB側の適用記録（`drizzle.__drizzle_migrations`テーブル）の3点を揃える**必要がある。

### Phase 0 — 読み取り専用の検証（承認不要・次セッションが最初にやること）

DB書き込みゼロ。既存の`pg`依存・`scripts/check-schema.cjs`の流儀で実施:

1. `SELECT * FROM drizzle.__drizzle_migrations ORDER BY id` — 本番が「適用済み」と認識している行数と`created_at`を確認する。**このテーブル自体が存在しない可能性もある**（過去に`migrate`が一度も走っておらず、全部direct applyだった場合）。存在しない場合はPhase 2の分岐Bへ。
2. `information_schema.columns` / `information_schema.tables` で、`0005`〜`0010`の内容（`locations.visibility`、`user_settings.livePresence*`、`event_participations`、`db_stats_snapshots`、`sponsor_cards`+`isSelfPromo`）が全部本番に在ることを確認する。
3. **ファイルに無いドリフトの検出**: `drizzle/schema/index.ts`のカラム一覧と`information_schema`を突き合わせ、「SQLファイル0000〜0010のどれにも書かれていないのに本番に在る/無い」差分がゼロかを確認する。`apply-location-schema-gaps.cjs`のような「隙間埋めスクリプト」が過去に走っている以上、**11本のファイルが本番の全履歴である保証はない**。差分が見つかったら、`IF NOT EXISTS`付きの`0011_reconcile.sql`として明文化してから先へ進む。

このPhase 0は後述の`scripts/db-journal-doctor.cjs`（P0成果物）としてスクリプト化し、使い捨てにしない。

### Phase 1 — リポジトリ側の修復（コミットのみ・DB無変更・承認不要）

1. **最新スナップショットの再生成**: 空の一時ディレクトリを`out`に指すscratch用drizzle configで`npx drizzle-kit generate`を1回実行 → 現行schema TS全体を表す`0000_snapshot.json`が得られる。これを`drizzle/migrations/meta/0010_snapshot.json`として配置し、`prevId`を既存`0003_snapshot.json`の`id`に合わせる（generateは最新スナップショットしか読まないため、0004〜0009のスナップショットは復元不要。`drizzle-kit check`が連鎖検証で文句を言う場合のみ調整）。
2. **journalへ追記**: `_journal.json`に`idx: 5`〜`10`のエントリを追加。`tag`はファイル名（拡張子なし）と完全一致、`when`は単調増加するミリ秒値（実適用日のタイムスタンプ推奨）。
3. Phase 0で差分が出ていれば`0011_reconcile.sql`＋対応journalエントリもここで追加。
4. コミット（`fix(db): _journal.jsonと本番DB実態の不整合を修復`）。

### Phase 2 — DB側のマーキング（**Stop And Ask・ユーザー承認必須**）

本番DBへの書き込みが発生するため、**実行前に必ずユーザーに手順と対象SQLを提示して承認を得ること**。CLAUDE.mdのDBスキーマ変更ルールの対象。

- **分岐A（`__drizzle_migrations`が存在し0000〜0004の記録がある場合・本命）**: `0005`〜`0010`（+あれば`0011`）に対応する行を`INSERT`する。`hash` = 各SQLファイル内容のsha256、`created_at` = journalの`when`と同値。drizzleのmigratorは`created_at`の大小で未適用を判定するため、この整合が本質。
- **分岐B（テーブルが無い/中身が信用できない場合・フォールバック=完全リベースライン）**: `0000`〜`0010`を`drizzle/migrations-archive/`へ退避（履歴の証跡として残す）、journalを空にし、現行schema TSから`0000_baseline`を新規generate、`drizzle.__drizzle_migrations`を作り直してbaseline 1行だけを適用済みマークする。手順が単純で決定的なので、分岐Aの検証で少しでも齟齬が出たら迷わずこちらに切り替えてよい。

### Phase 3 — 検証（承認不要）

1. `npx drizzle-kit generate` → 「No schema changes, nothing to migrate」相当の出力になること
2. `npx drizzle-kit migrate` → 適用0件で正常終了すること
3. `scripts/db-journal-doctor.cjs` → 全チェック緑

### 再発防止（このセクションの恒久ルール）

- **`scripts/apply-*.cjs`による本番直接ALTERを禁止**する。既存5本は`scripts/archive/`へ移動し、ファイル冒頭に「使用禁止・経緯はhandoff-2026-07-06.md 11章」と書く。スキーマ変更は必ず「schema TS変更 → `pnpm db:push`（=generate+migrate）→ 生成物コミット」の一本道。
- `run-drizzle.cjs`の`push`分岐の冒頭に`db-journal-doctor.cjs`を挟み、**不整合検出時はgenerateへ進まず非0終了**させる（次の誰かが同じ穴を掘れなくする）。

## 1.3 「型チェックは通るがランタイムで壊れる」の機械的検知（新規ライブラリゼロ）

### (a) ESM拡張子lint — `scripts/check-esm-imports.mjs`（P0）

本設計時点の実測で、**`server/`・`api/`配下に拡張子なし相対importが51箇所残存**している（`e8a8d9e4f`で直したのはsweep経路の9箇所だけ）。どれも「新しいコードが偶然そのモジュールをVercel Functions実行パスに乗せた瞬間」に`/api/sweep`と同じ死に方をする時限爆弾。

- Node標準（`fs`+正規表現）のみで実装: `server/`・`api/`配下の`.ts`を走査し、`from "./…"`/`from "../…"`が`.js`/`.json`で終わらないものを列挙して非0終了。
- 既存51箇所は**P0で一括修正してしまう**のが最も安い（機械的置換+`pnpm check`+本番sweep/health実測で閉じる。baseline方式で「既存は許容」とするより、爆弾の総数をゼロにする方がこのプロジェクトの体制に合う）。
- 組み込み: `package.json`の`"check": "tsc --noEmit && node scripts/check-esm-imports.mjs"`。既存の全ワークフロー・全エージェントの習慣（「pnpm check 0エラー必須」）にそのまま乗るので、**CLAUDE.mdの改訂すら不要**で全セッションに強制される。これがこの案の最大の利点。
- 参考: `tsconfig`の`moduleResolution: "nodenext"`化はTS2835を正式エラーにできるが、Expo/Metro側の解決規則と衝突するリスクがあるためP2の調査項目に留める（regexスクリプトで実害はゼロにできる）。

### (b) DBジャーナル・ドクター — `scripts/db-journal-doctor.cjs`（P0、§1.2 Phase 0の恒久化）

journal ⇔ SQLファイル ⇔ `__drizzle_migrations` ⇔ `information_schema`の4点照合。既存の`check-schema.cjs`と同じ`pg`+dotenvパターンで書ける。`run-drizzle.cjs`の`push`前フックとして組み込む（§1.2）。

### (c) `import()`直描画・依存配列事故の検知（P1、精度低め・通知のみ）

型2（参照同一性）は正規表現での完全検知は不可能。現実的な落とし所は既存`scripts/diff-check.mjs`の設定拡張:

- `diff-check.config.json`の`dangerFiles`に`lib/icons`・`hooks/use-live-presence`・`lib/lazy-heavy-components`を追加（触ったら警告が出る）
- `forbiddenWords`に相当する仕組みでdiff内の`= import(`をフラグ（`lazy(() => import(` は許可パターンとして除外）。誤検知が出うるので**ブロックせず警告**に留める。

### (d) キャッシュ層の一枚台帳 — `docs/CACHE-LAYERS.md`（P1）

型3の再発防止は仕組みより「地図」が効く。CDN（`CDN_CACHE_EPOCH`、現在値1）/ SW（`inject-sw-version.cjs`+`controllerchange`リロード）/ エントリHTML（`bust-entry-cache.cjs`の`?v=<sha>-<ts>`付与）/ `version.json`の4層について、「何をキャッシュし、何をキーに無効化され、どのスクリプトが責任を持ち、疑わしいときの確認コマンドは何か」を1ファイルの表にする。会議E案(4)「バージョニング方針の統一」は、**実装統一ではなくこの台帳化として採用**する（各層の仕組みは既に動いており、統一改修はデグレリスクの方が大きい）。

## 1.5 却下した案と理由

| 案 | 却下理由 |
|---|---|
| PRベース+人間レビュー必須（会議E批判役） | 制約8に正面衝突。無人・高速反復が体制の前提。代替として「pnpm checkへのゲート連結」で同等の機械的強制を得る |

# Part 2: UI/UX一貫性ガイドライン

## 2.1 SPEC A × SPEC B のトーン&マナー診断と、会議案の採否

**診断結果: 両SPECは既に統一されている。** SPEC Aは広告カードを「協賛の足あと」として切手/足あとカードの文法（`theme/tokens/`準拠）で定義し、SPEC Bとの整合コメント（§「足あと便SPECとの整合」）まで内蔵している。SPEC Bは郵便メタファー（足あと便・消印・停車地）で書かれている。つまり:

- **会議F発散役案1「コンテキストブリッジ層」: 実質採用（ただし新レイヤーは作らない）**。提案された「メタファーを通訳レイヤーにする」は、DESIGN.md＋両SPECのカード文法として**既に存在する**。やるべきは新造ではなく明文化 — 後述のGLOSSARY.mdに「新しいUI概念は郵便・旅の語彙系（便り/消印/切手/停車地/掲示板）から命名する」を核ルールとして書く。
- **案2「メタ認知誘導（声のラベリング）」: 半採用**。「これは協賛の声/これは記録」という薄いラベルは、`9eadea798`の「協賛/お知らせ」出し分けとして**既に実装済み**であり、この方向が正しかったことの実証でもある。これを**「声の分類法（voice taxonomy）」として体系化**する: **記録**（ユーザー自身の足あと。ラベルなし＝デフォルトの声）/ **お知らせ**（自社の声。isSelfPromo=true）/ **協賛**（第三者の声。景表法対応で法的必須）。今後増えるカード種はこの3分類のどれかに必ず属させる。一方、案2の「切り替わりにトランジションを入れる」は**不採用** — アニメーション総数を増やすだけで判別への寄与が薄く、制約4の精神（動きは最小限）に反する。
- **案3「ユーザー参加型統一」: 棄却**。ユーザー数が実測44行（locations）の段階で事後収束メカニズムを設計するのは完全な過剰設計。
- **fast役のデザインシステム総点検: 棄却**。`theme/tokens/`+DESIGN.mdが既にその役割を果たしており、抽象的な再整備はAI並行開発では「各エージェントが別解釈で再発明する」リスクの方が大きい。

## 2.2 訴求の衝突防止ルール（会議F批判役の指摘への具体回答）

批判役の懸念「チェックイン完了→広告→さらに訴求の畳み掛け」を、次の3ルールで潰す:

**ルール1: 1画面1商業訴求（Commercial Budget = 1）**
1つの画面状態で、商業的訴求（協賛カード・お知らせカード・プレミアムCTA）は**合計1枚まで**。行動訴求（シェアボタン・ログインCTA）はこれと別枠だが、同じく**1画面1つ**。「お知らせ」カードが自社機能を宣伝している場合、それは商業訴求としてカウントする（例: チェックイン完了画面に「シェア機能のお知らせカード」が出ているなら、同画面に別途プレミアム訴求を足してはならない）。

**ルール2: 訴求スタック順序の固定**
縦方向の情報順序を全画面で統一する: **①体験の余韻（結果・リプレイ・マッチ）→ ②記録の確認（履歴・統計）→ ③商業訴求（協賛/お知らせカード）→ ④行動訴求（シェア/ログインCTA）**。ただしSPEC Bの`/u/{slug}`は③が存在しない（後述）ため①→②→④。SPEC Aの「スクロールしないと見えない位置」原則はこの順序の実装形としてそのまま維持。
※例外: チェックイン完了画面で③と④が同居する場合、④（シェア）を③（スポンサー）より**上**に置く。拡散はこのアプリの成長エンジンであり、広告の下に沈めてはならない。

**ルール3: `/u/{slug}`の広告ゼロを昇格保証にする**
SPEC Aの配置表では「⏸ 当面入れない（P2で再検討）」だが、これを**「足あと便リプレイの開始〜完了までの体験中は永久禁止。ページ内への静的配置もSPEC A・本SPEC両方の改訂合意なしに解禁しない」**へ明文化して格上げする。理由: (1) ゲスト経路はdefer地雷（制約1）の巣、(2) Xから来た初見訪問者の第一印象がこのアプリのブランドそのもの、(3) SPEC B自体が既にログインCTA+シェアボタンの2訴求を持っており、ルール1の枠が埋まっている。

**クールダウン（軽量版）**: SPEC Aの既存キャップ（1セッション1枚・1日3回）は据え置きで十分。追加は1点のみ — **チェックイン完了画面でカードを表示した直後の遷移先（図鑑等）では、同一セッション中は次のカードを出さない**（SPEC A P1の図鑑枠実装時に、既存のクライアント側カウンタへ「直前表示からの画面遷移1回スキップ」を足すだけ）。

# Part 3: 並行開発運用の改善（軽量）

## 3.1 「気づかず並行push」を維持したまま事故を減らす機械的工夫

handoff文書の運用は**既に2回、実際に事故を検知している**（孤立ブランチの発見・journal不整合の発見）。これを捨てず、人の記憶に頼る部分だけ機械化する:

1. **`scripts/session-doctor.mjs`（新設・P0、Node標準のみ）**: handoff文書が「次モデルは必ずやれ」と書いている3点（`git fetch`+`HEAD..origin/main`確認 / `git worktree list`+孤立ブランチ検出 / 最新handoffファイルの所在表示）を1コマンドに束ね、加えて`db-journal-doctor`（Part 1）と未push コミットの有無を出力する。CLAUDE.mdに「セッション開始時に`node scripts/session-doctor.mjs`」の1行を追記。**チェックリストを読む義務を、コマンド1発に圧縮する**のが狙い。
2. **push直前の`git pull --rebase origin main`をディレクティブ4の手順に1行追記**（現状は「push→デプロイ確認」だけで、並行pushとの競合解決手順が暗黙）。コンフリクト時は自動マージせずhandoffに記録して停止、をルール化。
3. **push後diff-checkの通知化（P1）**: 既存Gate1はPR時のみ発火するが、この体制ではPRが立たない。`on: push, branches: [main]`で同じ`diff-check.mjs`を**落とさず警告として**走らせるworkflowを1本追加（`DIFF_BASE`は直前コミット）。dangerFiles（drizzle・vercel.json・auth系）に触れたpushが起きたことが、他セッション/ユーザーにActionsの履歴として残る。
4. **handoffの置き場所の規約化（P1）**: `docs/handoff/`ディレクトリに`YYYY-MM-DD.md`で置く規約に変更し、session-doctorが最新ファイルを自動表示。現行の`docs/handoff-2026-07-06.md`は移動。
5. **worktree干渉の既知対策の維持**: vitestのworktree除外（`aed0349`）のパターンを規約として明記 —「リポジトリ全体を走査するツール（テスト・lint・glob）を追加するときは`.claude/worktrees/**`除外を必ず入れる」。session-doctorのチェック項目にも入れる。

## 3.2 却下した案と理由

| 案 | 却下理由 |
|---|---|
| PRレビュー必須化・branch protection | 制約8。無人・高速反復が体制の核であり、人間レビューがボトルネックになった瞬間にこの開発速度は死ぬ。Gate1のprotection化はP2オプションとして温存に留める |
| ファイル/リソースのロック機構（セッション間排他） | ロックの解放忘れ・デッドロックという新しい事故クラスを持ち込む。実績のある「fetch確認+handoff+rebase」の徹底の方が安い |
| セッション間リアルタイム通信・調停レイヤー | 過剰。並行事故の実害は「成果物に気づかない」ことであり、session-doctor+handoffで解消できる |
| コミット毎の全自動デプロイ停止（手動承認制） | ディレクティブ4（同ターン内デプロイ）と矛盾。デプロイ検証の機械化（既存deploy-verify）を信頼する |
